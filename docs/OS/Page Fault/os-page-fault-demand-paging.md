# Page Fault와 Demand Paging

프로세스는 자신의 주소 공간에서 어떤 바이트든 자유롭게 읽고 쓸 수 있다고 생각합니다.
그런데 운영체제는 그 바이트가 실제 물리 프레임에 배정되어 있는 상태로 언제나 준비해 두지 않습니다.
매핑만 만들어 두고, 실제 접근이 일어나는 순간에 비로소 프레임을 붙이는 전략을 씁니다.
이 전략이 `demand paging`(요구 페이징) 이며, 그 실행 메커니즘이 `page fault`(페이지 폴트) 입니다.

## 페이지 폴트란 무엇인가

CPU가 가상 주소에 접근했을 때, MMU가 페이지 테이블을 따라가다가 문제를 만나면 페이지 폴트 예외를 발생시킵니다.
CPU는 현재 명령어 실행을 중단하고, 레지스터 상태와 폴트를 일으킨 가상 주소를 저장한 뒤, 커널의 `page fault handler` 로 제어를 넘깁니다.

```mermaid
sequenceDiagram
    participant UP as 유저 프로세스
    participant CPU as CPU/MMU
    participant K as 커널 핸들러
    participant DRAM
    UP->>CPU: VA에 load/store
    CPU->>CPU: PTE 조회
    alt PTE.P = 0 또는 권한 위반
        CPU->>K: Page Fault (VA, error_code)
        K->>K: 원인 분석
        alt 해결 가능 (정상 lazy alloc 등)
            K->>DRAM: 프레임 할당 / 로드
            K->>K: PTE 갱신 (P=1, PFN 설정)
            K-->>CPU: 복귀 → 원래 명령 재실행
        else 해결 불가능 (잘못된 접근)
            K->>UP: SIGSEGV 전달 → 종료
        end
    end
```

흥미로운 점은, 페이지 폴트가 "버그"가 아니라는 것입니다.
CPU와 커널 사이에 약속된 정상 신호입니다.
커널은 이 신호를 받아 상황에 맞게 처리하고, 문제가 없으면 중단된 명령어를 그대로 재실행합니다.
프로세스 입장에서는 아무 일도 없었던 것처럼 보입니다.

## 폴트의 세 가지 종류

여기서 중요한 점은 같은 "페이지 폴트"라도 원인에 따라 처리가 완전히 달라진다는 것입니다.
리눅스는 크게 세 종류로 구분합니다.

### Minor Fault — 디스크 접근 없이 해결

페이지의 내용은 이미 메모리 어딘가에 있지만, 해당 프로세스의 PTE에는 아직 연결되지 않은 경우입니다.
"데이터는 있는데 내 페이지 테이블이 모르는" 상태라고 할 수 있습니다.

```
예) fork 직후의 공유 페이지
    - 내용은 DRAM에 있음
    - PTE는 아직 "없음" 상태
    - fault 시 연결만 해주면 됨
```

디스크 I/O가 없으므로 수 μs 이내에 처리됩니다. 성능 측면에서 큰 부담이 아닙니다.

### Major Fault — 디스크에서 페이지를 읽어야 함

페이지의 내용이 디스크(실행 파일, 스왑 파일, `mmap` 된 파일)에 있어서 읽어 와야 하는 경우입니다.

```
예) 프로그램 로딩
    - .text 섹션이 아직 DRAM에 없음
    - 첫 실행 시 fault → ELF 파일에서 4 KB 읽어 프레임에 채움
    - PTE 갱신 후 복귀
```

수 ms 수준의 디스크 접근이 필요하므로 minor fault보다 수천 배 느립니다.
그래서 major fault 빈도가 시스템의 전반적 성능에 직접 영향을 미칩니다.

### Invalid Fault — 정말 잘못된 접근

폴트 주소가 어떤 VMA에도 속하지 않거나, 속하더라도 접근 권한이 맞지 않는 경우입니다.

```
예) NULL 역참조
    - 주소 0은 어떤 VMA에도 매핑되어 있지 않음
    - 커널은 이를 "버그"로 판정하고 SIGSEGV 전달
    - 프로세스 종료
```

커널은 "이 폴트는 정상적 lazy alloc이나 COW가 아니며, 정당화할 근거가 없다"고 결론 내리고 프로세스를 종료시킵니다.
우리가 보는 `Segmentation fault`가 바로 이 경로입니다.

| 종류         | 디스크 I/O | 처리 비용  | 결과                |
| ------------ | ---------- | ---------- | ------------------- |
| Minor Fault  | 없음       | ~μs       | 연결 후 재실행      |
| Major Fault  | 있음       | ~ms       | 로드 후 재실행      |
| Invalid Fault| 없음       | ~ms       | SIGSEGV, 프로세스 종료 |

## Demand Paging — "실제로 쓸 때만 준다"

요구 페이징은 이 폴트 메커니즘을 전략적으로 활용하는 설계입니다.
커널은 할당이 요청될 때마다 실제 프레임을 붙이는 대신 다음과 같이 처리합니다.

1. 가상 주소만 예약합니다 (VMA 등록).
2. 페이지 테이블 엔트리는 만들지 않거나, `P=0` 상태로만 만들어 둡니다.
3. 프로세스가 그 주소에 실제로 접근하는 순간에야 폴트를 통해 프레임을 배정합니다.

이 지연의 결과는 상당합니다.

- `malloc`으로 1 GB를 요청했지만 1 KB만 실제로 썼다면, 물리 DRAM은 1 KB만 소비됩니다.
  나머지 가상 주소는 PTE가 `P=0`인 상태로 남아 있을 뿐 프레임이 붙지 않습니다.
- 실행 파일이 100 MB여도, 실제 실행하는 코드가 5 MB뿐이라면 그 5 MB만 major fault로 메모리에 올라옵니다.
- `fork` 직후 자식 프로세스는 부모의 주소 공간을 복제한 것처럼 보이지만, 실제로는 페이지 내용을 공유한 채로 시작합니다.
  쓰기가 일어나는 페이지만 새 프레임을 받습니다 (COW).

```mermaid
flowchart TD
    A["malloc(1GB)"] --> B[커널이 VMA만 등록<br/>PTE는 P=0]
    B --> C{프로세스가 주소 X에 쓰기}
    C --> D[Page Fault]
    D --> E[커널이 실제 프레임 1개 배정]
    E --> F[PTE 갱신, P=1]
    F --> G[명령 재실행 → 쓰기 성공]
    G --> H{또 다른 페이지에 쓰나?}
    H -- yes --> C
    H -- no --> I["실제 사용 프레임 = 실제 접근한 페이지 수 × 4KB"]
```

이 전략은 세 가지 이점을 제공합니다.

- DRAM 사용 효율: 실제 필요한 만큼만 소비됩니다.
  거대한 가상 주소 공간이 실제 비용 없이 유지됩니다.
- 프로그램 시작 시간 단축: 실행 파일 전체를 미리 메모리에 올릴 필요가 없습니다.
  첫 명령어만 폴트로 가져오면 실행이 시작됩니다.
- 쓰지 않는 자원은 끝내 할당되지 않습니다: 오류 경로 코드, 예외 처리 루틴, 쓰지 않는 정적 배열은 폴트를 한 번도 일으키지 않고 프로세스가 끝납니다.

## 첫 접근의 지연: cold start

요구 페이징은 단점도 가집니다. "첫 접근은 느리다"는 점입니다.
막 시작한 프로세스가 코드의 새로운 부분을 실행할 때마다 major fault가 한 번씩 발생하고, `malloc`으로 받은 영역을 처음 쓸 때마다 minor fault가 한 번씩 발생합니다.
그래서 성능 측정은 `warm-up` 이후의 상태에서 해야 의미가 있습니다.
또한 지연에 민감한 시스템은 의도적으로 페이지를 미리 터치해서 폴트를 미리 발생시키기도 합니다 (`mlock`, 일부러 한 바이트씩 쓰기).

## 정리

페이지 폴트는 에러가 아니라 하드웨어와 커널이 약속한 통신 채널입니다.
커널은 이 채널을 통해 "실제 접근이 일어날 때만 프레임을 준다"는 게으른 전략을 구현합니다.
Minor·major·invalid 세 종류의 폴트를 상황에 맞게 처리함으로써, 거대한 가상 주소 공간을 실제 DRAM 용량을 훨씬 초과해 제공할 수 있습니다.
없는 것을 있는 척하는 일이 이렇게 정교한 예외 처리 시스템으로 가능해졌다는 사실이 가상 메모리 설계의 핵심 통찰입니다.
