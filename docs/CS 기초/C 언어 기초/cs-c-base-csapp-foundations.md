# C로 자료구조를 구현하기 전에 — CSAPP 관점의 기초 정리

자료구조 문제는 "알고리즘" 문제처럼 보입니다. 그런데 C로 직접 구현하려고 책상에 앉으면 상황이 달라집니다. 알고리즘이 맞아도 세그폴트가 납니다. 출력은 맞는데 메모리 누수가 생깁니다. 원소 1개일 때만 이상하게 죽습니다.

이 글은 Linked List·Stack·Queue·Binary Tree·BST를 C로 구현할 때 반복해서 발목을 잡는 부분을 CSAPP의 1장(시스템)·3장(기계 수준 프로그램)·9장(가상 메모리) 관점에서 다시 풀어 정리한 문서입니다. 한 권의 책을 압축하는 것이 목적이 아니라, 실제 문제를 풀다가 막히는 지점을 시스템 관점에서 설명하는 것이 목적입니다.

## 1. 왜 자료구조 문제에 시스템 지식이 필요한가

자료구조 문제를 C로 풀 때 같이 따라오는 것은 다음과 같습니다.

- 메모리 위에 자료구조를 어떻게 놓는가
- 포인터가 실제로 무엇을 가리키는가
- 함수 호출이 끝난 뒤 어떤 값이 남고 어떤 값이 사라지는가
- `malloc/free`를 언제 해야 하는가
- 세그폴트가 왜 나는가

즉 "정답 함수"만 쓰는 것이 아니라 자료구조 개념, C 포인터, 메모리 모델, 디버깅 감각이 함께 들어갑니다.

## 2. CSAPP 1장 관점 — 프로그램이 실행된다는 것

1장은 시스템 전체를 크게 보는 장입니다. 실제 문제 풀이에 필요한 요지는 다음과 같습니다.

- C 소스코드는 바로 실행되지 않습니다.
- 컴파일과 링크를 거쳐 실행 파일이 됩니다.
- 실행 파일은 메모리에 올라가서 CPU가 실행합니다.
- 프로그램은 OS, 메모리, 컴파일러, 입출력 환경과 연결되어 돕니다.

### 소스코드에서 실행 파일까지

우리가 작성하는 건 `.c` 파일입니다. 하지만 실제 실행되는 건 컴파일 결과물입니다.

```bash
gcc -g my_list.c -o my_list
```

이 명령이 하는 일은 "파일 하나를 돌린다"가 아닙니다.

- 문법을 검사합니다.
- 필요한 라이브러리를 연결합니다.
- 실행 가능한 프로그램을 만듭니다.

그래서 에러가 날 때 문법 문제인지, 타입 문제인지, 링크 문제인지, 실행 중 메모리 문제인지를 구분하는 게 중요합니다.

### 경고를 무시하면 안 되는 이유

C는 경고를 무시해도 실행 파일이 만들어지는 경우가 있습니다. 그런데 포인터와 동적 할당을 다루면 경고가 실제 버그로 이어질 가능성이 높습니다.

대표적인 경고 원인:

- 잘못된 포인터 타입
- return 누락
- 선언 없는 함수 호출
- 형 변환 없이 주소를 잘못 사용

"컴파일은 됐으니 됐다"가 아니라 경고가 없는 상태를 먼저 만드는 습관이 중요합니다.

### 시스템 전체 흐름을 보는 이유

문제를 풀면서 자주 생기는 착각은 "코드만 맞으면 된다", "출력만 맞으면 된다"입니다.

실제로는 출력이 맞아도 메모리 누수가 있을 수 있고, 몇 케이스는 맞아도 빈 리스트에서 세그폴트가 날 수 있고, 잘못된 포인터 접근이 우연히 지금만 안 터질 수 있습니다. 시스템 관점은 "왜 이 코드가 위험한가"를 보는 데 필요합니다.

## 3. CSAPP 3장 관점 — 함수·포인터·스택 프레임

3장 관점에서 중요한 핵심은 다음과 같습니다.

- 함수 호출과 반환
- 스택 프레임
- 값 전달
- 포인터를 통한 간접 접근
- 구조체와 배열의 메모리 배치
- 재귀 호출

### 함수 호출과 스택

함수를 부르면 끝이 아닙니다. 호출할 때마다 실행 문맥이 저장됩니다. 지역변수, 매개변수, 돌아갈 위치 정보가 스택에 쌓입니다.

왜 중요한가:

- 재귀 함수가 왜 동작하는지 이해할 수 있습니다.
- 지역변수 주소를 바깥에 넘기는 게 왜 위험한지 감이 잡힙니다.
- 함수가 끝나면 지역변수는 더 이상 안전하지 않다는 것을 이해하게 됩니다.

### 값 전달이 기본이라는 점

C는 기본적으로 값 전달입니다.

```c
void f(int x) {
    x = 100;
}
```

이 함수는 바깥 변수를 바꾸지 못합니다. `x`는 복사본이기 때문입니다.

그런데 포인터를 넘기면 상황이 달라집니다.

```c
void f(int *p) {
    *p = 100;
}
```

함수는 "원본 값을 직접 가리키는 주소"를 받습니다.

자료구조 문제와 연결되는 지점:

- head를 바꿔야 하는 함수
- 노드 내용 자체를 수정하는 함수
- 리스트를 분할하거나 합치는 함수

이런 함수들이 왜 포인터나 이중 포인터를 쓰는지 이해해야 합니다.

### 구조체와 포인터 접근

Linked List, Tree, BST는 전부 구조체 기반입니다.

```c
typedef struct _listnode {
    int item;
    struct _listnode *next;
} ListNode;
```

여기서 중요한 점:

- `next`는 다음 노드 "값"이 아니라 다음 노드 "주소"입니다.
- 자료구조를 바꾼다는 건 결국 주소 연결을 바꾸는 일입니다.

그래서 `cur->next = newNode;` 같은 코드는 "연결 구조를 다시 배선한다"는 의미로 읽는 것이 좋습니다.

### 배열, 포인터, 문자열

SQL 처리기 같은 문자열 중심 문제에서는 배열·문자열 감각도 필요합니다.

- 입력 버퍼
- SQL 토큰 분리
- 문자열 비교
- 파일에서 읽은 줄 처리

는 전부 배열과 포인터 감각을 같이 씁니다. 문자열은 결국 `char` 배열이고, 포인터로 순회할 수 있습니다.

즉 자료구조 문제는 연결 포인터 중심이고, 문자열 처리 쪽은 문자열 포인터와 버퍼 관리도 중요해집니다.

### 재귀가 왜 자연스러운가

트리 문제를 보면 재귀가 자연스럽습니다.

- 동일한 구조를 왼쪽 서브트리와 오른쪽 서브트리에 반복 적용
- 높이 계산
- 노드 개수 세기
- 합 계산

재귀가 잘 맞는 이유는 트리 자체가 자기 유사 구조이고, 현재 노드 처리 + 하위 문제 처리로 나눌 수 있기 때문입니다.

하지만 순회 순서가 중요하면 반복 + stack/queue가 더 자연스러울 때도 있습니다.

- BST level-order: queue
- iterative preorder/inorder/postorder: stack

## 4. CSAPP 9장 관점 — 가상 메모리와 동적 할당

9장은 실전에서 바로 터지는 장입니다.

### 메모리 영역을 구분해서 보기

최소 아래 네 구역을 알고 있으면 좋습니다.

- text: 실행 코드
- data / bss: 전역변수, 정적변수
- stack: 함수 호출, 지역변수
- heap: `malloc`으로 받은 동적 메모리

자료구조 노드는 대부분 heap에 생성합니다. 함수가 끝난 뒤에도 계속 살아 있어야 하고, 노드 개수가 실행 전에 고정되지 않기 때문입니다.

### 왜 지역변수 노드를 연결하면 안 되는가

다음과 같은 코드는 위험합니다.

```c
ListNode node;
head = &node;
```

함수 안에서 이 코드를 했다면 함수가 끝난 뒤 `node`는 안전하지 않습니다.

- 잠깐만 필요한 데이터는 stack
- 함수 밖까지 살아야 하는 노드는 heap

이 기준을 확실히 잡아야 합니다.

### 동적 할당의 책임

`malloc`을 썼으면 결국 누군가는 `free`해야 합니다.

문제 풀이에서 꼭 따져볼 것:

- 누가 노드를 만들었는가
- 그 노드를 나중에 누가 정리하는가
- 삭제 시 그 자리에서 `free`해야 하는가
- 프로그램 종료 시 전체 정리 함수가 있는가

자료구조 과제에 `removeAllItems`, `removeAll`, `removeAllItemsFromQueue`, `removeAllItemsFromStack`가 반복되는 이유가 여기 있습니다.

### 메모리 누수

메모리 누수는 프로그램이 끝날 때 운영체제가 회수해 주는 것과 별개로, 코드 수준에서는 여전히 버그입니다.

누수가 자주 생기는 상황:

- 노드를 새로 만들고 연결 안 함
- 삭제하면서 `free`를 안 함
- 기존 구조를 새 구조로 옮기고 원래 구조를 안 정리함
- Queue/Stack 임시 구조를 만들고 정리 안 함

### 세그멘테이션 폴트

세그폴트는 "허용되지 않은 메모리 접근"으로 이해하면 됩니다.

자주 나오는 원인:

- `NULL`인데 `node->next` 접근
- 빈 큐인데 dequeue
- 빈 스택인데 pop
- 이미 free한 노드 다시 접근
- 리스트 연결을 잘못 끊어서 엉뚱한 주소 접근

즉 세그폴트는 자료구조 개념 문제이면서 동시에 시스템 문제입니다.

## 5. 문제 유형별로 필요한 시스템 감각

### Linked List

문제 핵심:

- 삽입 위치 찾기
- 이전 노드와 현재 노드 추적
- head 바뀌는 경우 처리
- 삭제 시 연결과 free 순서 맞추기

필요한 시스템 감각:

- 포인터 변수 이동과 원본 구조 변경 구분
- 이중 포인터 필요 여부 판단
- heap에 있는 노드를 안전하게 해제하기

대표 실수:

- 첫 노드 처리 누락
- 마지막 노드 처리 누락
- `size`만 바꾸고 연결 안 바꿈
- free 후 재접근

### Stack / Queue

문제 핵심:

- 삽입/삭제 위치가 정해져 있음
- FIFO, LIFO 성질 유지
- 원래 순서 보존 여부 판단

필요한 시스템 감각:

- 기존 리스트 함수를 재사용할 때 포인터 흐름 이해
- 임시 구조를 만들면 누가 정리하는지 생각
- 비어 있는 구조에서 꺼낼 때 방어 코드 넣기

### Binary Tree

문제 핵심:

- `NULL`이 base case
- 현재 노드와 하위 트리 처리 분리
- 재귀 호출 흐름 이해

필요한 시스템 감각:

- 재귀 호출마다 스택 프레임이 생긴다는 점
- 종료 조건이 없으면 무한 재귀 위험
- 순회 중 포인터가 `NULL`이 될 수 있다는 점

### Binary Search Tree

문제 핵심:

- BST 규칙 유지
- 순회 종류에 따라 stack/queue 선택

필요한 시스템 감각:

- queue는 level-order에 맞습니다.
- stack은 preorder/inorder/postorder iterative에 맞습니다.

즉 BST 문제는 자료구조 2개가 합쳐져 있습니다.

### 문자열·파일 처리 (SQL 처리기 등)

문자열 중심 문제는 자료구조 문제와 다르지만 기본은 같습니다.

- 입력 문자열 저장
- 토큰 분리
- 명령 종류 판별
- 파일에 기록
- 파일에서 다시 읽기

필요한 시스템 감각:

- 문자열 버퍼 관리
- 파일 I/O 흐름
- 고정 길이 버퍼와 오버플로우 주의
- 동적 메모리 쓸 경우 해제 책임 정리

## 6. 실제 문제를 풀 때 적용하는 순서

### 자료구조 관점

먼저 문제를 보고 아래를 적어 봅니다.

- 입력 구조는 무엇인가
- 출력 또는 목표 구조는 무엇인가
- 기존 노드를 이동하는가
- 새 노드를 만드는가
- 순서를 유지해야 하는가

### 포인터 관점

그 다음 아래를 봅니다.

- head/root가 바뀌는가
- 이전 노드를 잡아야 하는가
- 현재 노드 다음을 미리 저장해야 하는가
- `NULL` 체크가 어디 필요한가

### 메모리 관점

마지막으로 확인합니다.

- `malloc`이 필요한가
- `free`는 어디서 하는가
- 함수 종료 뒤에도 살아야 하는 데이터인가
- 임시 자료구조는 정리되는가

## 7. 디버깅 관점에서 보면 더 잘 보이는 것들

### 출력이 이상하면

- 연결 순서가 틀린지
- 삽입/삭제 위치가 틀린지
- 순회를 잘못했는지 봅니다.

### 아예 죽으면

- `NULL` 접근
- 빈 구조에서 pop/dequeue
- free 후 접근
- 재귀 종료 조건 누락

을 먼저 봅니다.

### 일부 케이스만 틀리면

- 빈 입력
- 원소 1개
- 첫 노드 / 마지막 노드
- 한쪽 자식만 있는 트리

를 따로 테스트합니다.

## 8. 꼭 설명 가능해야 하는 질문

- 왜 C에서는 포인터를 넘겨야 원본이 바뀌는가
- 왜 head를 바꾸는 함수는 이중 포인터가 필요한가
- 왜 노드는 heap에 만들어야 하는가
- 왜 삭제할 때 `free`와 연결 변경 순서가 중요한가
- 왜 트리는 재귀로 풀기 쉽고, 어떤 경우 stack/queue가 필요한가
- 왜 세그폴트는 단순 문법 문제가 아니라 메모리 접근 문제인가

## 9. 최소 실전 체크리스트

- 함수 호출 후 원본이 바뀌어야 하는지 확인했는가
- head/root 변경을 놓치지 않았는가
- `NULL` 체크가 있는가
- `malloc`한 메모리를 누가 `free`하는지 정했는가
- 빈 구조와 원소 1개 케이스를 테스트했는가
- 반복/재귀 중 어떤 방식이 더 자연스러운지 먼저 판단했는가

## 10. 정리

자료구조 문제는 자료구조 문제처럼 보이지만, C로 풀면 실제로는 시스템 문제이기도 합니다.

- 1장 관점은 "프로그램이 전체 시스템 안에서 실행된다"는 감각
- 3장 관점은 "함수·포인터·구조체·재귀가 메모리와 연결된다"는 감각
- 9장 관점은 "동적 메모리와 메모리 안전성을 직접 책임진다"는 감각

이 세 가지가 같이 잡혀야 C로 자료구조를 구현할 때 덜 흔들립니다.
