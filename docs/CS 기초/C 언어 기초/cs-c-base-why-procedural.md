# C가 절차적인 이유 — 하드웨어를 직역하는 언어

C++·C#·Java로 커리어를 시작한 입장에서 C는 늘 "낡은 언어"였습니다. 클래스도 없고, 가상 함수도 없고, 제네릭도 없고, 소유권도 없고, 예외도 없고. "있어야 할 것이 다 없는" 언어.

이번에 CSAPP 9장을 읽고 minidb를 구현하며, 이 인상이 한 단계 꺾였습니다. C는 "있어야 할 것이 빠진" 언어가 아니라 "다른 층위에서 추상화하는" 언어였습니다. 이 글은 그 생각을 정리해 나간 기록입니다.

## C 언어에 대한 첫인상

minidb 설계를 시작하며 C에게 거부감을 느낀 지점을 차례로 적어 둡니다.

### 다형성

"B+ tree의 내부 노드와 잎 노드를 같은 페이지 단위로 다루고 싶은데, 어떻게 타입 구분을 하지?"

머릿속에 떠오른 답은 당연히 다음과 같았습니다.

```cpp
class Node {
public:
    virtual void insert(int key) = 0;
};
class InnerNode : public Node { ... };
class LeafNode : public Node { ... };
```

C에는 없습니다. 함수 포인터 테이블을 구조체에 넣어 흉내 내는 방법이 있지만, "왜 이걸 이렇게 돌아서 해야 하나" 싶었습니다.

### 직렬화

"페이지를 디스크에 어떻게 쓰지? `FileOutputStream` 같은 건 없고..."

```java
objectOut.writeObject(node);
```

이 한 줄이 해 주던 일을 C에서 어떻게 하는가 막막했습니다.

### 자원 관리

"RAII로 `std::unique_ptr` 쓰면 free 걱정이 없는데, C는 매번 free를 써야 한다니..."

하나하나가 작은 짜증이었고, 처음 며칠은 "C는 왜 아직도 살아 있는가"를 속으로 투덜거리는 시간이었습니다.

## 전환점: page_type 태그

`page_type` 태그로 페이지 다형성을 구현한 순간 감각이 바뀌었습니다. 핵심은 이것입니다.

```c
typedef enum : uint32_t {
    PAGE_TYPE_FREE, PAGE_TYPE_META, PAGE_TYPE_HEAP,
    PAGE_TYPE_BPLUS_INNER, PAGE_TYPE_BPLUS_LEAF,
} PageType;

// 모든 페이지 구조체의 첫 4바이트가 이 태그
switch (*(PageType *)f->data) {
    case PAGE_TYPE_HEAP:  handle_heap((HeapPage *)f->data); break;
    case PAGE_TYPE_BPLUS_LEAF: handle_leaf((LeafPage *)f->data); break;
    ...
}
```

객체 지향 언어였다면 `Node` 베이스 클래스와 v-table로 풀었을 일입니다. 그런데 C의 방식은 단순하고, 중요한 차이가 있었습니다.

메모리 블록 자체가 자기 타입을 알립니다. 그리고 그 "자기 타입의 알림"이 디스크에도 그대로 기록됩니다.

## OOP와 C의 추상화 방식 차이

이 지점에서 제 생각이 뒤집혔습니다.

OOP의 다형성은 "같은 인터페이스를 가진 여러 구현"을 추상화합니다. `Animal::speak()` 뒤에 `Dog`, `Cat`, `Cow`가 숨습니다. 이 추상화의 단위는 "객체의 identity"입니다. 이 놈이 Dog이냐 Cat이냐. 그리고 객체는 서로 다른 힙 주소에 흩어져 존재합니다.

C의 페이지 기반 설계의 추상화는 다릅니다. 추상화의 단위는 "메모리 블록"입니다. "이 4 KB 블록은 지금 어떻게 해석되어야 하는가"가 질문입니다. 블록은 모두 같은 크기이고 같은 자리에 배치됩니다. 다형성은 같은 바이트를 서로 다른 스키마로 읽는 성질입니다.

```
 OOP:       [Animal A]  [Animal B]  [Animal C]     각기 다른 주소/크기
             ↓ speak()   ↓ speak()   ↓ speak()      vtable 로 디스패치

 C page:    [  Block 0  ][  Block 1  ][  Block 2  ] 같은 크기, 같은 배열
             첫 4B: type   첫 4B: type   첫 4B: type   switch 로 디스패치
```

두 방식이 같은 문제에 대한 두 가지 답처럼 보이지만, 실은 다른 문제를 풀고 있습니다.

- OOP는 "복잡한 객체 그래프"를 모델링할 때 강합니다. 현실의 개념들 (주문, 고객, 상품, 할인 정책)을 클래스로 나누고, 각 클래스가 자기 책임을 가집니다.
- C 페이지 모델은 "균일한 블록의 배열"을 모델링할 때 강합니다. 메모리 페이지, 디스크 섹터, 네트워크 패킷, 캐시 라인. 모두 같은 크기의 덩어리가 연속으로 늘어서 있고, 그 덩어리들에 해석을 얹는 문제.

OS, 데이터베이스, 네트워크 스택, 임베디드 제어는 후자의 문제입니다. 그래서 이 분야가 C로 쓰이는 것이 자연스럽습니다.

## 직렬화에서도 같은 원리

"직렬화를 어떻게 하지"도 같은 관점에서 풀렸습니다. OOP 모델에서는 객체가 힙에 흩어져 있으니 디스크에 쓰려면 모아 줄여 담는 변환이 필요합니다. `writeObject()`가 하는 일이 그것입니다.

C 페이지 모델에서는 메모리 표현과 디스크 표현이 애초에 같은 비트 패턴입니다. `memcpy` 하나가 직렬화입니다.

이것이 의미하는 바는 OOP의 직렬화 설계가 객체 그래프와 평면 바이트 스트림 사이의 간극을 메우는 일이었다는 것입니다. C는 그 간극을 처음부터 없게 설계합니다. 대신 받아들여야 할 제약이 있습니다 — 포인터 금지, 패딩 금지, 엔디안 고정.

## 자원 관리와 C의 설계 철학

"매번 free를 써야 한다"는 거부감도 관점이 바뀌었습니다. C의 메모리 관리가 번거롭지만 예측 가능하다는 걸 체감했습니다.

GC 언어에서 "아 이거 free 안 해도 되니 편하다"라고 느끼던 자리는, 사실 GC가 뒤에서 일을 하고 있는 자리였습니다. 이 일이 얼마나 큰 일인지 몰랐을 뿐. GC가 돌 때마다 프로세스가 잠깐 멈추고, 메모리 사용량이 예측 불가능해지고, 데이터베이스처럼 latency-critical한 시스템에선 이것이 문제가 됩니다.

C는 개발자에게 번거로움을 주는 대신 런타임을 없앱니다. malloc/free가 호출되는 순간만이 메모리를 만지는 순간입니다. 그 외엔 프로그램이 프로그래머가 쓴 그대로 돌아갑니다. 이 성질은 OS 커널, 실시간 시스템, 디스크 기반 엔진 같은 곳에서 대체 불가능한 가치를 가집니다.

## 의도된 최소주의

지금까지 정리해 보니 C에 대한 첫인상 "있어야 할 것이 없다"가 완전히 틀렸습니다.

- 클래스가 없다 → 구조체 + 태그 + switch로 다형성을 표현. 객체 그래프보다 메모리 블록의 해석을 추상화 단위로 쓴다.
- 예외가 없다 → 리턴 코드로 에러를 명시. 에러 경로가 코드에 눈에 보이게 남는다.
- 제네릭이 없다 → 매크로, void 포인터, 코드 생성으로 대응. 컴파일 타임의 복잡성이 낮다.
- GC가 없다 → 수동 메모리 관리. 예측 가능한 성능.
- RAII가 없다 → `goto cleanup;` 패턴. "가다가 실패하면 정리 코드로 점프"라는 명확한 제어 흐름.
- 네임스페이스가 없다 → `prefix_foo()` 네이밍 컨벤션. 링커 레벨에서 깔끔하다.

이 각각이 "불편한 빈자리"가 아니라, 다른 종류의 추상화를 허용하기 위한 선택이었다는 것을 이번에 깨달았습니다. 그리고 그 선택들이 연관되어 있다는 것도. 예를 들어 GC가 없다는 선택이 구조체를 메모리 위에서 그대로 `memcpy`할 수 있게 만듭니다 (GC 언어의 객체는 GC 헤더 때문에 이게 안 됩니다). 클래스가 없다는 선택이 패딩을 통제할 수 있게 만듭니다 (v-table 포인터가 없으니까). 이 선택들이 모여 "메모리 레이아웃을 프로그래머가 완전히 통제한다"는 속성을 만들고, 이 속성이 OS 커널과 DB 엔진과 네트워크 스택을 가능하게 합니다.

## 절차적 프로그래밍의 재평가

처음 C를 만나면 "절차적"이라는 말이 "원시적"처럼 들립니다. OOP가 등장하기 전의 방식. 진화의 이전 단계.

이번에 깨달은 건 그게 아니었습니다. 절차적 언어는 "상태의 변화를 순차적으로 기술"하는 언어입니다. 메모리 위의 데이터가 시간을 따라 어떻게 변하는가를 그대로 코드로 씁니다. 이것은 하드웨어가 실제로 돌아가는 방식과 정확히 일치합니다. CPU가 명령어를 하나씩 실행하고, 메모리 주소의 값이 순차적으로 바뀌고, 레지스터가 갱신됩니다. 절차적 기술은 이 실재를 은폐하지 않고 그대로 드러냅니다.

OOP는 이 위에 "개체"라는 층을 쌓습니다. 레지스터와 메모리 주소 대신, 서로 메시지를 주고받는 객체들의 세계로 문제를 번역합니다. 이 번역은 복잡한 도메인 모델링에 아주 강력합니다. 하지만 하드웨어 단에 가까운 문제에서는 이 번역이 오히려 방해가 됩니다. 번역 비용이 성능과 예측 가능성을 먹기 때문입니다.

C가 여전히 커널과 DB 엔진과 임베디드에서 쓰이는 이유는, 그 분야들이 번역을 원치 않는 분야이기 때문입니다. 하드웨어의 단위로 생각하고 싶고, 메모리의 레이아웃을 직접 정하고 싶고, 실행 순서를 코드에 그대로 반영하고 싶습니다.

## 결론

"C는 왜 절차적인가?"에 대한 답은 이것입니다.

C는 하드웨어를 모델링하기 위해 설계된 언어이고, 하드웨어는 본질적으로 절차적입니다. 메모리는 바이트의 배열이고, CPU는 명령을 순차적으로 실행합니다. C의 절차성은 이 물리적 실재를 프로그래밍 모델로 직역한 결과입니다.

그 실재를 은폐하지 않는 대가로 얻는 것이 있습니다. 예측 가능성. 내가 쓴 코드가 실행 시간과 메모리 측면에서 정확히 무엇을 할지 알 수 있습니다. 이 성질이 OS와 DB와 드라이버와 실시간 제어의 기반 언어로 C가 지금도 자리를 지키는 이유였습니다.

OOP는 C보다 더 진화한 언어가 아니라, 다른 문제를 풀기 위해 다른 층위에서 추상화하는 언어였습니다. 이 둘 사이에 우열은 없습니다. 풀려는 문제가 다를 뿐입니다. 제가 OOP에 익숙해져 있었다는 건 제가 지금까지 풀어 온 문제가 OOP가 잘 푸는 문제들이었다는 뜻이기도 했습니다. 이번에 풀어 본 문제 — 디스크 기반 SQL 엔진 — 는 OOP가 잘 푸는 문제가 아니었습니다. 그래서 C가 더 깔끔한 답을 냈습니다.

도구에 선호를 가진 것이 아니라 문제에 선호를 가진 것이었다는 게, 가장 큰 배움입니다.
