# 메모리 버그의 근본 원인: 포인터의 메타데이터 부재

C와 C++은 메모리를 다루는 데 강력한 자유를 줍니다. 그런데 그 자유가 깨지는 순간 프로그램이 침묵하거나 거짓말을 합니다.

"Segmentation fault"부터 "왜 이 값이 이상하게 바뀌었지?"까지. 처음 이 주제를 공부하며 깨달은 점은, 모든 메모리 버그가 결국 한 문장으로 귀결된다는 것입니다.

**C의 포인터는 자기가 가리키는 대상에 대해 아무것도 모릅니다.**

이 무지가 있기에 할당기·컴파일러·운영체제는 버그를 사전에 막을 방법이 없습니다.

## 포인터에 없는 메타데이터

C의 포인터는 기계어 수준에서 64비트 정수 하나입니다. 이 값은 다음 중 어느 것도 내재적으로 담고 있지 않습니다.

- 가리키는 블록의 크기는 얼마인가
- 가리키는 블록이 여전히 유효한가 (이미 free된 건 아닌가)
- 가리키는 바이트가 초기화되었는가
- 그 위치에 어떤 타입이 놓여 있는가

자바스크립트의 객체 핸들이나 러스트의 `Box<T>`에는 이 정보의 일부가 실립니다. C는 그렇지 않습니다. 포인터는 그냥 숫자입니다.

런타임에 그 숫자를 역참조하는 순간, 그 주소의 바이트를 "요청한 타입의 값"으로 해석합니다. 거기 있는 것이 프로그래머가 저장한 값인지, 다른 무언가가 덮어쓴 값인지, 해제된 블록의 잔해인지 — 컴파일된 코드는 구분할 수 없습니다.

이 구조적 사실이 메모리 버그의 모든 얼굴입니다.

## 메모리 버그 7가지 유형

### 1. Out-of-Bounds (OOB)

배열이나 버퍼의 경계를 넘는 접근입니다.

```c
int arr[10];
arr[10] = 0;   // 인덱스 0..9만 유효
```

컴파일러는 대부분 경고도 주지 않습니다. 런타임에도 메모리는 그냥 그 주소에 쓰기를 허용합니다.

우연히 그 자리에 있던 다른 변수가 덮어써집니다. 그 변수가 언제 이상해지는지는 실행 패턴마다 달라서 버그가 재현되지 않는 것처럼 보입니다.

### 2. Use-After-Free (UAF)

이미 `free`된 블록에 계속 접근합니다.

```c
char *p = malloc(16);
free(p);
printf("%s\n", p);   // 이 블록은 이미 free list에 들어갔을 수 있다
```

할당기는 free 블록의 payload 자리에 free list 포인터를 저장합니다. 따라서 free된 블록을 읽으면 그 자리에 있던 할당기의 내부 메타데이터를 읽게 됩니다.

운이 좋으면 segfault로 죽습니다. 나쁘면 이상한 문자열이 출력된 뒤 한참 뒤에 크래시가 납니다.

심지어 그 블록이 즉시 다른 `malloc` 호출에게 재할당되면, 두 포인터가 같은 메모리를 공유하게 되어 예측 불가능한 상호 파괴가 시작됩니다.

### 3. Double Free

한 블록을 두 번 해제합니다.

```c
free(p);
free(p);   // p가 여전히 같은 주소를 가리키고 있으면
```

할당기는 두 번째 `free`가 무효임을 모를 수 있습니다. free list의 같은 노드를 두 번 집어넣게 되어 리스트 자체가 사이클을 가진 상태가 됩니다. 다음 `malloc`이 이상하게 작동하거나, free list 순회가 무한 루프에 빠집니다.

### 4. Memory Leak

`free`를 잊어 블록이 영원히 할당된 채 남는 상태입니다.

```c
while (1) {
    void *p = malloc(1024);
    // free(p) 빠짐
}
```

당장은 죽지 않습니다. 시간이 지나 RSS가 부풀어 오르다가 결국 OOM killer에 의해 종료됩니다. 장기 실행 서버에서 가장 은밀한 버그입니다.

### 5. Dangling Pointer

가리키는 대상이 이미 사라진 포인터입니다.

```c
char *p;
{
    char local[16];
    p = local;
}
// p는 이제 사라진 스택 프레임을 가리킨다
```

스택 프레임이 되감기거나 블록이 `free`된 직후에도 포인터 자체는 여전히 같은 값을 가집니다. 그것을 역참조하면 UAF의 친척이 됩니다.

### 6. Uninitialized Read

`malloc`으로 받은 메모리나 지역 변수를 초기화 없이 읽습니다.

```c
int x;
if (x > 0) { ... }   // x는 이전 스택 프레임의 잔해를 담고 있을 수 있다
```

`malloc`이 반환하는 바이트는 이전 소유자가 쓴 잔해 그대로입니다 (`calloc`이 아니라면). 유효한 메모리이기에 프로그램은 크래시 없이 계속 실행됩니다. 하지만 결과는 실행마다 다르고 재현되지 않습니다.

### 7. Stack Overflow

스택이 자신의 VMA 끝을 넘어서 자라는 상태입니다.

```c
void recurse(void) { recurse(); }   // 무한 재귀
int huge[1 * 1024 * 1024];          // 지역 배열로 4 MB 잡기
```

리눅스는 스택 VMA 바로 아래에 가드 페이지를 두어 스택 넘침을 `page fault`로 잡아냅니다. 하지만 한 프레임이 한 번에 여러 페이지를 뛰어넘을 정도로 크면 가드를 건너뛰어 다른 VMA를 침범할 수 있습니다.

## 공통 원인: 포인터의 메타데이터 부재

일곱 가지 얼굴을 한 발짝 떨어져 보면, 원인은 셋 중 하나의 조합입니다.

**경계를 모른다**
포인터가 자기 블록의 크기를 몰라서 OOB·stack overflow가 가능합니다.

**수명을 모른다**
포인터가 자기 대상의 생사를 몰라서 UAF·double free·dangling pointer가 가능합니다.

**내용을 모른다**
포인터가 자기 대상의 초기화 여부를 몰라서 uninitialized read가 가능합니다.

컴파일러·할당기·OS는 이 세 가지 정보 모두를 각자의 장부에 가집니다. 그러나 그 장부는 포인터 자체와 연결되어 있지 않습니다. 포인터는 여전히 단지 숫자이고, 역참조는 그 숫자가 가리키는 바이트를 읽거나 쓸 뿐입니다.

## 디버깅 도구의 개입 방식

이 구조적 한계를 우회하기 위해, 외부 도구는 포인터가 없는 메타데이터를 따로 관리합니다.

### AddressSanitizer (ASan) — Shadow Memory

ASan은 컴파일 타임에 모든 메모리 접근 앞뒤로 섀도우 메모리 검사 코드를 삽입합니다. 섀도우 메모리는 실제 메모리 8바이트당 1바이트를 가집니다. 이 1바이트가 "이 8바이트 중 어디까지 유효한가"를 표현합니다.

```
 실제 주소 공간  :  [ payload 8B ] [ payload 8B ] [ payload 8B ]
 섀도우 메모리   :  [ 1 byte ]     [ 1 byte ]     [ 1 byte ]

 섀도우 값:  0   → 8바이트 모두 유효
             k   → 앞쪽 k바이트만 유효 (k=1..7)
             -1  → 이 영역은 free 되었음(poisoned)
```

`malloc` 시 ASan은 블록 전후에 **레드존(redzone)** 을 만들고 그 자리의 섀도우를 poisoned로 표시합니다. 경계를 넘어서는 접근이 있으면 섀도우 조회에서 걸려 즉시 보고됩니다. `free` 시 블록 전체의 섀도우를 poisoned로 바꿔 UAF도 잡습니다.

```mermaid
flowchart LR
    CODE["주소에서 로드/저장"] --> CHECK[ASan이 addr>>3의 섀도우 조회]
    CHECK -->|섀도우=0이고 권한 OK| OK[정상 진행]
    CHECK -->|중독됨 또는 부분적 유효| ERR[오류 보고 후 종료]
```

ASan이 있으면 버그가 나자마자 보고됩니다. 런타임 오버헤드는 수 배 수준입니다.

### Valgrind — 해석 실행

Valgrind의 Memcheck는 프로그램을 VM 위에서 실행하며 모든 메모리 접근을 추적합니다. 컴파일 재빌드 없이도 쓸 수 있지만, ASan보다 훨씬 느립니다.

### Guard Page, MALLOC_PERTURB_

glibc의 일부 디버그 기능은 `free` 시 바이트를 특정 패턴으로 채워 UAF가 읽을 때 눈에 띄게 합니다.

## 정리

C의 메모리 버그는 프로그래머의 부주의가 아니라 언어 설계의 구조적 선택이 만든 결과입니다.

포인터가 메타데이터를 들고 다니지 않는 한, 경계·수명·초기화는 프로그래머의 머릿속에서만 추적됩니다.

ASan·Valgrind 같은 도구는 포인터 바깥에 그 정보를 외주로 두어 버그를 잡습니다. 그러나 언어 수준에서 이 문제를 근본적으로 푸는 쪽은 Rust나 관리형 런타임처럼 포인터에 의미를 되돌려 주는 설계입니다.

C로 짓는 한 이 한계는 상수입니다. 할당기와 도구와 규율로 보완할 수밖에 없습니다.
