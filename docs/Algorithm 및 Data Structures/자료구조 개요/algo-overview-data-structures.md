# 자료구조 분류 — 네 가지 문제 축으로 보는 선택 기준

자료구조를 공부하다 보면 이름과 특성은 외우게 되지만, "지금 내 문제에는 무엇을 써야 하는가" 라는 선택의 감각은 잘 잡히지 않습니다. 이 글은 자료구조를 "문제의 성격" 기준으로 네 축에 나눠 두고, 각 축 안에서 대표 구조가 어떤 제약을 받아들이고 어떤 성능을 얻는지 정리합니다.

## 자료구조란 무엇인가

자료구조는 컴퓨터 메모리에 데이터를 배치하고 조직화하는 방법입니다. 데이터가 "물건" 이라면 자료구조는 그 물건을 담는 선반·상자·파일첩에 해당합니다.

자료구조를 고를 때 중요한 것은 단순 저장이 아니라 네 가지 연산의 비용입니다.

- 읽기 (Search)
- 추가 (Insert)
- 삭제 (Delete)
- 수정 (Update)

같은 데이터를 보관해도 어떤 구조에 담느냐에 따라 각 연산의 시간 복잡도가 달라지고, 데이터 규모가 커질수록 그 차이는 기하급수적으로 벌어집니다.

## 선택의 두 축 — 시간과 공간

자료구조 선택은 결국 두 가지 효율 사이에서 타협하는 일입니다.

- 시간 복잡도 (Time Complexity) — 각 연산을 얼마나 빨리 수행할 수 있는가.
- 공간 복잡도 (Space Complexity) — 같은 데이터를 저장할 때 메모리를 얼마나 덜 쓰는가.

100만 권의 책이 무작위로 쌓인 창고와, 가나다순으로 정리된 창고에서 특정 책을 찾는 속도 차이를 생각하면 직관적입니다. 정렬된 창고는 더 많은 공간과 유지 비용을 요구하지만 탐색은 훨씬 빠릅니다. 자료구조의 선택도 이와 같은 구조입니다.

## 네 가지 문제 축

자료구조를 "어떤 문제에 쓰이는가" 로 보면 네 개의 축으로 정리됩니다.

| 축 | 문제의 성격 | 대표 자료구조 |
|---|---|---|
| 순서 | 데이터의 순서·방향이 의미를 가진다 | 배열, 스택, 큐, 데크, 연결 리스트 |
| 관계 | 계층·포함 관계가 있다 | 트리, 이진 탐색 트리, 자가균형 트리, B-Tree, 힙 |
| 연결 | 다대다 관계망이 있다 | 그래프 |
| 속도 | 키로 즉시 찾아야 한다 | 해시 테이블, 딕셔너리, 맵, 셋 |

이 분류는 엄격한 정의가 아니라 선택을 돕는 프레임입니다. 같은 구조가 여러 축에 걸치는 경우도 많습니다. 하지만 "내 문제는 어느 축에 가까운가" 를 먼저 떠올리면 후보군이 크게 좁혀집니다.

## 순서가 중요한 문제

### 배열 (Array)

비슷한 물건을 나란히 연속된 사물함에 보관하는 구조입니다.

- 데이터를 메모리에 연속해 저장하므로 시작 주소와 인덱스만 있으면 임의 위치에 O(1) 로 접근할 수 있습니다.
- 대신 크기가 고정적이고, 같은 타입의 원소만 담을 수 있습니다.

C 배열과 파이썬 리스트는 같은 "배열" 이라는 이름을 쓰지만 내부가 다릅니다.

| 구분 | C 배열 (원시 배열) | 파이썬 리스트 (동적 배열 + 참조) |
|------|----------------------|-------------------------------|
| 사물함에 저장되는 것 | 실제 값 | 값을 가리키는 주소(포인터) |
| 타입 제한 | 동일 타입만 | 혼합 가능 |
| 크기 | 선언 시 고정 | 동적으로 확장 |

C 배열은 메모리에 값이 그대로 연속해 붙어 있고, 파이썬 리스트는 포인터의 배열입니다. 이 차이가 캐시 친화성, 할당 비용, 직렬화 가능성 등 많은 후속 설계 선택을 가릅니다.

### 스택 (Stack)

접시를 쌓아 올리는 구조입니다. 삽입·삭제가 모두 "맨 위" 에서만 일어납니다.

- LIFO (Last In First Out) — 나중에 들어온 것이 먼저 나갑니다.
- 주요 연산: `push` (삽입), `pop` (꺼내기), `peek` (맨 위 확인).
- 활용 예: 함수 호출 스택, 브라우저 뒤로 가기, 괄호 짝 검사.

### 큐 (Queue)

줄을 서는 구조입니다. 먼저 들어온 것이 먼저 나갑니다.

- FIFO (First In First Out).
- 주요 연산: `enqueue` (삽입), `dequeue` (꺼내기).
- 활용 예: 프린터 대기열, 은행 번호표, 서버 요청 처리.

### 데크 (Deque, Double-Ended Queue)

양쪽 끝에서 모두 삽입·삭제할 수 있는 구조입니다. 큐의 앞·뒤 양쪽에서 연산할 수 있어 스택과 큐의 상위집합에 해당합니다.

- 주요 연산: `append`·`appendleft`·`pop`·`popleft`. 파이썬에서는 `collections.deque` 가 양쪽 끝 연산을 모두 O(1) 로 제공합니다.
- 리스트의 `pop(0)` 은 O(n) 이지만 deque 의 `popleft()` 는 O(1) 이므로, 큐를 구현할 때는 리스트보다 deque 가 훨씬 효율적입니다.
- 활용 예: 슬라이딩 윈도우, 브라우저 앞·뒤로 이동 기록, 작업 스케줄링.

### 연결 리스트 (Linked List)

각 사물함이 다음 사물함의 주소를 함께 적어 두는 구조입니다.

- 데이터를 메모리 어딘가에 흩어 저장하고, 각 노드가 다음 노드의 주소를 가리키는 방식으로 연결됩니다.
- 배열과 달리 크기가 유동적이고, 중간 삽입·삭제가 빠릅니다. 대신 특정 위치를 찾으려면 처음부터 순서대로 따라가야 해 임의 접근은 O(n) 입니다.
- 활용 예: 음악 플레이어 재생 목록, 실행 취소(Undo) 기록.

## 관계가 중요한 문제

### 트리 (Tree)

뿌리(Root) 에서 가지가 뻗어 나가는 계층 구조입니다.

- 하나의 루트 노드에서 시작해 자식 노드들이 연결되며 뻗어 나갑니다. 사이클은 없습니다.
- 가족관계도나 폴더 구조처럼 계층적 데이터를 다루기 위해 쓰입니다.
- 주요 개념: 루트, 부모/자식, 리프(Leaf), 깊이(Depth), 높이(Height).
- 활용 예: 파일 시스템, 조직도, HTML DOM.

### 이진 탐색 트리 (Binary Search Tree, BST)

"왼쪽은 작은 값, 오른쪽은 큰 값" 규칙을 지키는 이진 트리입니다.

- 모든 노드에서 왼쪽 서브트리의 값은 자신보다 작고, 오른쪽 서브트리의 값은 자신보다 큽니다.
- 이 규칙 덕분에 이진 탐색처럼 절반씩 범위를 줄여 가며 탐색할 수 있어, 균형이 유지되면 탐색 복잡도는 O(log n) 입니다.
- 단점: 정렬된 순서로 삽입이 들어오면 한쪽으로 치우쳐 선형 구조(최악 O(n)) 로 떨어집니다.

### 자가균형 트리 (Self-Balancing Tree)

BST 의 편향 문제를 해결하기 위해 고안된 트리입니다. 삽입·삭제 시 자동으로 트리 높이를 균형 있게 조정합니다.

- AVL 트리 — 왼쪽·오른쪽 서브트리의 높이 차이가 최대 1 을 넘지 않도록 유지합니다. 균형 조건이 엄격해 탐색 속도가 빠른 대신, 회전(Rotation) 연산이 자주 발생합니다.
- 레드-블랙 트리 — 노드에 빨강/검정 색상을 부여하고 규칙으로 균형을 유지합니다. AVL 보다 균형 조건이 느슨해 삽입·삭제가 빠르고, 실무에서 가장 널리 쓰입니다. Java `TreeMap`, C++ STL `map`, Linux 커널 스케줄러 등에 쓰입니다.

### B-Tree / B+Tree

한 노드가 여러 개의 키와 자식을 가질 수 있는 다분기 트리입니다.

- 이진 트리가 자식을 최대 2 개만 갖는 것과 달리, B-Tree 는 한 노드에 여러 키를 저장하고 여러 자식을 가집니다.
- 트리의 높이가 낮아지므로 탐색에 필요한 I/O 횟수가 줄어듭니다. 디스크 기반 저장소처럼 "한 번의 접근 비용이 비싼" 환경에 적합합니다.
- B+Tree 는 B-Tree 의 변형으로, 실제 데이터는 리프 노드에만 저장하고 리프 노드끼리 연결 리스트로 이어 둡니다. 덕분에 범위 탐색이 매우 효율적입니다.
- 활용 예: 관계형 DB 인덱스(MySQL InnoDB 등), 파일 시스템.

### 힙 (Heap)

항상 가장 큰 값 혹은 가장 작은 값이 뿌리에 오도록 유지되는 완전 이진 트리입니다.

- 최대 힙은 부모가 자식보다 크거나 같고, 최소 힙은 반대입니다.
- 극값(최댓값 또는 최솟값) 을 O(1) 로 읽고 O(log n) 에 꺼낼 수 있어 우선순위 큐의 기반 구조로 쓰입니다.
- 활용 예: 우선순위 큐, 다익스트라 최단 경로, 힙 정렬.

## 연결이 중요한 문제

### 그래프 (Graph)

노드(정점) 와 노드를 잇는 간선(Edge) 으로 이루어진 일반 관계 구조입니다.

- 트리와 달리 사이클이 있을 수 있고, 간선에 방향이 있을 수도(방향 그래프) 없을 수도(무방향 그래프) 있습니다.
- 지도 경로 탐색, 친구 추천처럼 복잡한 관계망을 표현하기 위한 구조입니다.
- 주요 탐색 방법: BFS(너비 우선), DFS(깊이 우선).
- 활용 예: 내비게이션 경로 탐색, SNS 친구 추천, 네트워크 라우팅.

## 속도가 중요한 문제

### 해시 테이블 (Hash Table)

키를 해시 함수에 넣으면 저장 위치(인덱스) 가 계산되는 구조입니다.

- 탐색·삽입·삭제가 평균 O(1) 로 매우 빠릅니다.
- 서로 다른 키가 같은 위치로 매핑되는 해시 충돌(Collision) 이 발생할 수 있으며, 이를 어떻게 해결하는지(체이닝, 개방 주소법, 리해싱 정책) 가 실제 성능을 좌우합니다.
- 활용 예: 로그인 세션 관리, 캐시 스토어, DB 인덱싱.

### 딕셔너리 / 맵 / 셋 (Dictionary / Map / Set)

해시 테이블을 기반으로 각 언어가 제공하는 고수준 자료구조입니다.

| 이름 | 설명 | 대표 언어 |
|------|------|----------|
| 딕셔너리 | 키-값 쌍 저장 | Python `dict` |
| 맵 | 딕셔너리와 같은 개념, 순서 보장 버전도 있음 | Java `HashMap`, JavaScript `Map` |
| 셋 | 중복 없는 값의 집합 | Python `set`, Java `HashSet` |

세 가지 모두 내부적으로 해시 테이블을 사용하므로 평균 O(1) 접근이 가능합니다. 활용 시나리오는 중복 제거(Set), 빠른 키 기반 조회(Map/Dict), 카운팅(Dict) 으로 자연스럽게 갈라집니다.

## 선택 기준 정리

자료구조 선택은 다음 질문들로 좁혀 나갈 수 있습니다.

- 데이터에 순서가 있는가, 있다면 어느 쪽 끝에서 주로 연산하는가. → 배열·스택·큐·데크·연결 리스트 중 선택.
- 계층이나 정렬된 범위 탐색이 필요한가. → 트리·BST·자가균형 트리·B-Tree.
- 극값을 반복적으로 꺼내야 하는가. → 힙.
- 다대다 관계를 표현해야 하는가. → 그래프.
- 키로 즉시 조회해야 하는가. → 해시 테이블과 그 파생 구조.

자료구조의 이름을 외우는 것보다, 이 질문들을 먼저 던지는 습관이 구조 선택의 정확도를 크게 높여 줍니다. 같은 문제를 여러 구조로 풀어 보고 각각의 시간·공간 비용을 비교해 보는 것이, 결국 자료구조를 "쓸 줄 아는" 단계로 넘어가는 가장 빠른 길이었습니다.
