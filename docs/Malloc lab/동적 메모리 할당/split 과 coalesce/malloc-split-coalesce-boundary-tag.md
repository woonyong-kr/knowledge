# Split 과 Coalesce — 경계 태그가 만들어 내는 O(1) 병합

할당기의 본문은 `malloc` 과 `free` 의 두 함수지만, 내부에서 반복해서 호출되는 핵심 연산은 split 과 coalesce 입니다. 둘은 대칭이면서 서로의 반대로 작동합니다. split 은 큰 블록을 나눠 요청에 맞추는 연산, coalesce 는 해제된 블록을 인접 블록과 합쳐 외부 단편화를 줄이는 연산입니다. 이 두 연산을 빠르게 해 주는 장치가 경계 태그(boundary tag) 와 prev-allocated 비트입니다.

## Split — 언제 쪼갤 수 있는가

`malloc` 요청이 들어와 찾은 가용 블록이 요청 크기보다 충분히 크면 split 합니다. "충분히 크다" 의 기준은 쪼갠 나머지가 최소 블록 크기 이상이라는 것입니다. 최소 블록 크기는 보통 헤더 8B 와 최소 페이로드 8B 를 합친 16B 이지만, Explicit Free List 라면 prev/next 포인터 공간도 필요하므로 더 커집니다.

```
 split 조건
   가용 블록 크기 - 요청 블록 크기  ≥  최소 블록 크기

 예 1) 가용 64B, 요청 24B
   나머지 = 64 - 24 = 40B ≥ 16B → split 한다
   앞쪽 24B 를 할당, 뒤쪽 40B 에 새 헤더를 달아 free list 에 삽입

 예 2) 가용 32B, 요청 24B
   나머지 = 32 - 24 = 8B < 16B → split 못 한다
   32B 전체를 할당 → 8B 내부 단편화 발생
```

split 을 하지 않으면 내부 단편화, split 을 적극적으로 하면 외부 단편화 가능성이 커집니다. 이 트레이드오프가 할당자 설계의 실전 감각을 결정합니다.

## Coalesce — 네 가지 케이스

`free` 호출 시 현재 블록을 가용으로 바꾸고, 인접한 블록이 이미 가용이면 하나로 합칩니다. 이전(PREV) 과 다음(NEXT) 의 할당 상태 조합에 따라 네 가지 케이스가 나옵니다. 편의상 현재 해제된 블록을 `B` 라 부르겠습니다.

```
 Case 1 : PREV 할당, NEXT 할당
   합칠 상대 없음 → B 만 가용으로 표시, free list 에 삽입
   결과 : [PREV 할당][B 가용][NEXT 할당]

 Case 2 : PREV 할당, NEXT 가용
   B + NEXT 를 합친다
   B 헤더에 합친 크기 기록, NEXT 를 free list 에서 제거, 합친 블록을 삽입
   결과 : [PREV 할당][B+NEXT 가용]

 Case 3 : PREV 가용, NEXT 할당
   PREV + B 를 합친다
   PREV 헤더에 합친 크기를 갱신 (PREV 는 이미 free list 소속)
   결과 : [PREV+B 가용][NEXT 할당]

 Case 4 : PREV 가용, NEXT 가용
   PREV + B + NEXT 셋을 모두 합친다
   PREV 헤더에 합친 크기를 갱신, NEXT 를 free list 에서 제거
   결과 : [PREV+B+NEXT 가용]
```

네 케이스 모두 분기만 다를 뿐 연산 자체는 동일한 모양입니다. 문제는 이 판단을 얼마나 빨리 할 수 있느냐입니다.

## 네 케이스가 모두 O(1) 인 이유

판단에 필요한 정보가 세 가지 있습니다. (1) 다음 블록의 상태, (2) 이전 블록의 상태, (3) 이전 블록의 크기. 세 가지 모두 상수 시간에 확인 가능합니다.

- 다음 블록 상태 — 현재 블록 헤더에서 크기를 읽어 다음 헤더 주소를 계산하고, 그 헤더의 할당 비트를 봅니다. O(1).
- 이전 블록 상태 — 현재 블록 헤더의 P 비트(bit 2) 를 읽습니다. O(1).
- 이전 블록 크기 — 현재 블록 헤더의 바로 앞 워드를 읽습니다. 그 자리에 이전 블록의 풋터(헤더 복사본) 가 있으므로 거기서 크기를 얻습니다. O(1).

이 세 정보가 모두 상수 시간이므로 coalesce 네 케이스 전부 O(1) 입니다. "경계 태그의 O(1) 마법" 이라 부르는 것이 바로 이 지점입니다.

## 최적화 — 할당 블록의 풋터를 없앤다

풋터는 사실 모든 블록에 필요하지 않습니다. coalesce 는 이전 블록이 "가용" 일 때만 합치므로, 이전 블록이 할당 상태라면 크기를 알 필요가 없습니다.

이 관찰에서 최적화가 나옵니다. 할당 블록의 풋터는 제거하고, 현재 블록 헤더의 P 비트로 이전 블록 상태만 유지합니다.

```
 규칙
   가용 블록 → 풋터 유지(coalesce 에 필요)
   할당 블록 → 풋터 제거(P 비트만으로 충분)

 판단 흐름
   P = 1 → 이전 블록 할당, 합칠 필요 없음, 이전 풋터가 없어도 무방
   P = 0 → 이전 블록 가용, 합쳐야 할 수 있음, 이전 풋터에서 크기를 읽음
```

이 최적화로 할당 블록마다 4~8 바이트를 아낍니다. 블록 수가 많을수록 누적 절약량이 커집니다. glibc 의 ptmalloc 이 이 방식을 실제로 사용하고, CSAPP 교재에서도 "최적화된 경계 태그" 로 소개합니다.

## 정리

split 과 coalesce 는 할당기의 실전 연산입니다. split 은 블록 크기를 요청에 맞추며 단편화 종류를 선택하는 결정이고, coalesce 는 해제 시 연속 공간을 복구하는 주기적 청소입니다. 풋터(boundary tag) 는 이전 블록의 경계를 상수 시간에 읽기 위한 장치이고, prev-allocated 비트는 그 풋터를 할당 블록에서는 생략하기 위한 최적화입니다. 할당기 성능의 대부분이 이 네 개 개념의 정교한 맞물림에서 나옵니다.
