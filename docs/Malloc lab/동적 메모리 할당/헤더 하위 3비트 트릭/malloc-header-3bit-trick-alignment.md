# 헤더 하위 3비트 트릭 — 정렬이 만들어 준 공짜 비트

할당기는 블록 크기와 상태를 함께 저장해야 합니다. 그 두 정보를 따로 두면 블록마다 여분 바이트가 필요해집니다. 실전 할당기는 정렬 규칙을 역이용해 두 정보를 한 워드에 담습니다. 이 기법이 "헤더 하위 3비트 트릭" 입니다.

## 블록 크기가 8 의 배수여야 하는 이유

x86-64 에서 가장 큰 기본 타입은 double 과 포인터로 둘 다 8바이트입니다. CPU 는 데이터를 읽을 때 주소가 자연 정렬(natural alignment) 되어 있으면 한 번의 메모리 접근으로 끝내지만, 정렬이 깨지면 두 번 접근이 필요하거나 하드웨어에 따라 예외가 납니다.

`malloc` 이 반환하는 주소는 그 블록 안에 어떤 기본 타입이 올지 모르기 때문에 항상 8바이트 정렬이어야 합니다. 블록 시작 주소를 8의 배수로 잡고, 블록 크기도 8의 배수로 만들면 다음 블록 역시 자동으로 8바이트 정렬이 됩니다. 이 연쇄로 힙 전체의 정렬이 지켜집니다.

블록 크기가 8의 배수라는 말은 이진수로 하위 3비트가 항상 `000` 이라는 뜻입니다. 그 3비트는 크기 정보에 쓰이지 않으므로 "공짜 비트" 로 재활용할 수 있습니다.

## 하위 3비트에 담는 플래그

표준적인 구성은 다음과 같습니다.

```
 헤더 (8 바이트)

 63                               3 │  2  │  1  │  0  │
 ┌─────────────────────────────────┬─────┬─────┬─────┐
 │      블록 크기 (8-aligned)        │  P  │  ?  │  A  │
 └─────────────────────────────────┴─────┴─────┴─────┘

 A (bit 0) — 현재 블록 할당 상태
   1 = 할당, 0 = 가용
 ? (bit 1) — 구현 재량의 예약 비트
   glibc 에서는 mmap 으로 받은 블록인지 표시
 P (bit 2) — 이전 블록 할당 상태
   1 = 이전 블록 할당, 0 = 이전 블록 가용
```

각 비트는 용도가 정해져 있습니다. `A` 는 이 블록의 현재 상태, `P` 는 이전 블록의 상태입니다. `P` 가 중요한 이유는 coalesce 때문입니다. 인접한 이전 블록과 합칠 수 있는지를 이 비트 하나로 상수 시간에 판단합니다(풋터를 거치지 않고 헤더만으로).

크기를 읽을 때는 마스킹으로 하위 3비트를 지웁니다.

```c
size_t size    = header & ~0x7;   // 하위 3비트 제거
int    is_alloc = header &  0x1;  // bit 0 만 추출
int    prev_alloc = (header & 0x4) >> 2; // bit 2 만 추출
```

별도의 구조체가 없어도 한 워드 읽기로 크기와 두 가지 상태를 모두 얻습니다.

## 검증 — 헤더값 0x19 는 어떤 블록인가

구체적인 헤더값으로 해석해 보겠습니다.

```
 헤더 = 0x19 = 0b 0001 1001

 하위 3비트 : 001
   bit 0 (A) = 1 → 현재 블록 할당
   bit 1     = 0
   bit 2 (P) = 0 → 이전 블록 가용

 블록 크기 = 0x19 & ~0x7
          = 0x19 & 0xFFFFFFF8
          = 0x18
          = 24 바이트

 확인: 0x19 = 25 에서 하위 비트 1 을 빼면 24 = 8 × 3 이 됩니다.
```

다음 블록 헤더의 주소는 현재 헤더 주소에 블록 크기를 더한 값입니다.

```
 현재 헤더 주소 = hp
 다음 헤더 주소 = hp + 24

 예: hp = 0x1000 이면
   현재 블록        : 0x1000 ~ 0x1017 (24B)
   다음 블록 헤더   : 0x1018
```

한 줄 요약은 이렇습니다. 헤더 `0x19` 는 "이전 블록은 가용, 현재 블록은 할당, 블록 크기는 24B" 의 세 정보를 압축해 담고 있습니다.

## 이 트릭이 아끼는 것

별도의 상태 필드를 두지 않으므로 블록마다 바이트를 절약합니다. 블록이 수천·수만 개라면 그 절약이 쌓여 유의미해집니다. 또 하나의 이점은 상태와 크기를 한 번의 로드로 가져온다는 점입니다. 캐시 라인 하나에서 모든 판단을 끝낼 수 있어 분기 예측과 메모리 접근 양쪽에 유리합니다.

## 정리

정렬은 제약처럼 보이지만 할당기 입장에서는 공짜 저장 공간이기도 합니다. 8바이트 정렬이 만들어 준 하위 3비트에 할당 상태·이전 블록 상태·구현 재량 플래그를 담으면 헤더 하나로 모든 메타데이터가 해결됩니다. "제약이 최적화 공간을 만든다" 는 할당기 설계의 전형적인 장면입니다.
