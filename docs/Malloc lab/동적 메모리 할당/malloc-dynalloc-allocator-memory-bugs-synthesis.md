# 할당기와 메모리 버그의 연결 — 단편화부터 힙 오염까지 한 줄로 잇기

할당기를 다루는 글이 흩어지면 각 개념이 서로 어떻게 연결되는지가 흐려집니다. 실제로는 단편화 → 자료구조 → 헤더 설계 → 동작 → 위험 까지가 한 줄기입니다. 이 글은 그 줄기를 한 번에 잇는 요약본입니다. 각 장에 세부 글이 따로 있으므로 여기서는 전체 흐름과 연결만 짚습니다.

## 한 줄 요약

Heap allocator 는 단편화 문제를 줄이기 위해 복잡한 자료구조와 정책을 사용하고, C 는 그 과정의 안전성 검사를 거의 프로그래머 책임으로 남겨 두기 때문에 메모리 버그가 쉽게 발생합니다.

이 문장이 할당기와 메모리 버그를 한 덩어리로 묶는 근거입니다. 문제 → 구조 → 동작 → 위험의 네 단계가 여기에 모두 들어 있습니다.

## 1. 문제 — allocator 가 먼저 풀어야 하는 것은 단편화다

`malloc` 이 실패하는 이유는 메모리가 모자라서가 아니라, 사용 가능한 형태로 남아 있지 않아서입니다.

```
 외부 단편화
   [16B 할당][48B 가용][16B 할당][32B 가용]
   총 가용 = 48 + 32 = 80B
   요청    = 60B
   → 48B 도 32B 도 60B 미만, 중간에 할당 블록이 있어 합칠 수도 없음
```

내부 단편화는 정렬과 최소 블록 규약의 부산물이고, 외부 단편화는 시간에 따른 할당·해제 패턴의 부산물입니다. 둘의 원인이 다르니 전략도 다르고, 할당기 설계의 큰 줄기가 바로 이 두 단편화를 각각 어떻게 억제하느냐입니다.

## 2. 구조 — 탐색 비용을 줄이기 위한 bin

모든 블록을 매번 순회하는 방식으로는 실전 처리량이 나오지 않습니다. 자료구조 선택이 탐색 비용을 바로 정합니다.

```
 Implicit Free List   : O(전체 블록 수)
 Explicit Free List   : O(가용 블록 수)
 Segregated Free List : 해당 bin 만 탐색 → ≈ O(1)
```

Segregated 구조의 핵심은 "탐색 집합을 크기 구간으로 좁히는 것" 입니다. bin 안에서 아무 블록이나 고르면 best-fit 근사를 상수 시간에 얻습니다. glibc 의 ptmalloc·jemalloc·tcmalloc 대부분이 이 조합을 기본으로 씁니다.

## 3. 구조 — 한 워드에 크기와 상태를 담는 헤더 트릭

블록 크기를 8 바이트 정렬로 강제하면 하위 3비트가 항상 `000` 이 됩니다. 이 공짜 비트에 할당 상태를 담아 헤더 하나로 모든 메타데이터를 압축합니다.

```
 [ 블록 크기 | P | - | A ]
 A (bit 0) : 현재 블록 할당 여부
 P (bit 2) : 이전 블록 할당 여부  → coalesce 판단에 사용
```

헤더값 `0x19` 를 예로 들면 `size = 24B`, `A = 1 (할당)`, `P = 0 (이전 가용)` 이 한 번의 마스킹으로 얻어집니다. 정렬 규칙이 만들어 준 저장 공간을 재활용하는 전형적인 시스템 프로그래밍 기법입니다.

## 4. 동작 — split·coalesce·placement 의 맞물림

할당 순간마다 세 결정이 동시에 일어납니다.

- split — 큰 블록을 요청 크기에 맞게 나눕니다. 잔여가 최소 블록 크기 미만이면 쪼개지 않고 그대로 할당합니다. split 을 안 하면 내부 단편화, 지나치게 하면 외부 단편화가 커집니다.
- coalesce — 해제된 블록의 앞뒤 가용 블록과 합칩니다. 네 가지 케이스(PREV·NEXT 의 상태 조합) 모두 헤더 P 비트와 풋터의 조합으로 O(1) 에 처리됩니다.
- placement — 어떤 가용 블록을 고를지 결정합니다. First Fit · Next Fit · Best Fit 각각 트레이드오프가 다르고, segregated + 클래스 내 first-fit 의 조합이 현대 할당기의 기본 선택입니다.

세 연산이 맞물려 할당자의 단편화 억제와 처리량이 동시에 결정됩니다.

## 5. 위험 — free() 는 올바른 포인터를 가정한다

`free()` 가 잘못된 포인터를 검증하지 않는 이유는 성능 때문입니다. 매 호출마다 "이 주소가 정말 malloc 이 반환한 블록 시작 주소인가" 를 확인하면 상수 시간이 무너집니다. 그래서 C 표준은 잘못된 free 를 UB 로 남기고 검증 책임을 프로그래머에게 넘겼습니다.

이 설계는 세 가지 대표 사례에서 바로 위험이 드러납니다.

```
 double free
   같은 블록이 free list 에 두 번 등록됨
   이후 malloc 두 번이 같은 주소를 반환 → heap corruption

 free(p + 1)
   사용자 데이터를 헤더로 착각해서 읽음
   엉뚱한 크기·상태 해석 → free list 파괴

 int x; free(&x)
   스택 주소를 힙 블록처럼 해석
   힙과 스택 메타데이터가 뒤섞임 → 리턴 주소 오염 가능
```

요점은 세 경우 모두 allocator 가 신뢰하는 전제(포인터가 블록 시작이고, 앞에 진짜 헤더가 있다) 를 깨뜨린다는 것입니다. 성능을 위한 신뢰가 위험의 문을 여는 구조입니다.

## 6. 위험 — 메모리 버그는 구조적 결과다

메모리 버그는 서로 별개의 실수 목록이 아닙니다. 모두 allocator 가 유지하는 경계·헤더·free list 를 깨뜨리는 동일한 사건으로 수렴합니다.

- Buffer Overflow — 다음 데이터나 다음 블록 헤더를 덮어씀.
- Use-After-Free — 이미 free list 에 들어간 블록을 다시 사용.
- Double Free — 같은 블록이 free list 에 두 번 등록됨.
- Memory Leak — 할당된 메모리를 가리키는 포인터를 잃어버림.
- Uninitialized Read — `malloc` 이 메모리를 0 으로 초기화하지 않아 이전 데이터 흔적을 읽음.

아래 짧은 코드 조각은 두 가지 버그를 동시에 만듭니다.

```c
int **A = malloc(n * sizeof(int));   // 버그 1: sizeof(int*) 가 맞다
// ... 사용 ...
A[n] = NULL;                          // 버그 2: 유효 범위는 0 ~ n-1
```

첫 줄은 `sizeof(int)` 로 할당해 크기를 부족하게 잡았고, 마지막 줄은 경계 밖 쓰기로 다음 블록의 헤더를 덮어쓸 수 있습니다. 겉보기엔 단순한 실수지만 구조적으로는 allocator 의 경계·헤더 가정 두 가지를 동시에 깨뜨리는 사건입니다.

## 그러므로 알 수 있는 것

네 단계를 한 문장씩으로 요약하면 다음과 같습니다.

- 문제 — 할당기의 핵심 과제는 메모리 부족이 아니라 단편화 관리다.
- 구조 — 크기별 free list 와 헤더 3비트 트릭이 탐색 비용과 메타데이터 비용을 동시에 줄인다.
- 동작 — split·coalesce·placement 의 맞물림이 단편화 억제와 처리량을 결정합니다.
- 위험 — C 는 이 구조를 빠르게 유지하려 검사를 최소화했고, 그래서 잘못된 포인터 하나가 힙 전체를 오염시킬 수 있습니다.

allocator 의 여섯 개 개념은 흩어져 있는 듯 보여도 실은 같은 축선 위에 있습니다. 그 축선을 따라 내려오면 메모리 버그까지 자연스럽게 도달합니다. 그래서 **allocator 를 이해해야 메모리 버그도 이해할 수 있다** 는 한 줄이 성립합니다.
