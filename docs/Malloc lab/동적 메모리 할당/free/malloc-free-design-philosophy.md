# free() 의 설계 철학 — 왜 에러를 알려주지 않는가

`free()` 의 시그니처는 놀랍도록 단순합니다. 포인터 하나를 받고 값을 돌려주지도 않습니다. 반환값이 `void` 라는 점은 "성공/실패를 알려주지 않겠다" 는 선언이기도 합니다. 잘못된 포인터를 넘겨도 함수는 조용히 힙 구조를 뭉개고 돌아옵니다. 이 기이한 설계는 C 의 철학을 가장 솔직하게 드러냅니다.

## 왜 UB 로 남겨 두었는가

`free()` 가 잘못된 포인터를 검증하려면 매 호출마다 "이 주소가 정말 `malloc` 이 반환한 블록 시작 주소인가" 를 확인해야 합니다. 그 검증을 하려면 힙 전체 블록 목록을 뒤지거나 주소 범위와 메타데이터의 일관성을 검사해야 하고, 이는 상수 시간을 쉽게 벗어납니다.

C 의 설계 원칙은 "프로그래머를 신뢰하고, 런타임 오버헤드를 최소화한다" 입니다. 그 원칙에 맞추기 위해 표준은 잘못된 `free` 를 오류가 아닌 Undefined Behavior 로 남겨 뒀습니다. 검증 비용을 프로그래머의 책임으로 넘긴 것입니다.

반환값이 `void` 인 것도 같은 배경입니다. 성공/실패를 알려 주려면 판단이 필요하고, 판단은 검증을 요구합니다. C 는 그 비용을 거부했습니다.

## 잘못된 포인터가 만드는 네 가지 위험

잘못된 포인터로 `free` 를 호출하면 다음 중 하나 이상의 결과가 나옵니다.

첫째, 힙 메타데이터 파괴입니다. `free` 는 전달받은 포인터 바로 앞에 있는 헤더(블록 크기·할당 비트·prev 정보 등) 를 읽습니다. 그 자리에 실제 헤더가 없으면 엉뚱한 값을 헤더로 해석하고, 존재하지도 않는 블록을 해제한 것처럼 free list 를 조작합니다.

둘째, 다른 블록 침범입니다. 깨진 free list 때문에 다음 `malloc` 이 이미 사용 중인 블록을 반환할 수 있습니다. 두 포인터가 같은 메모리를 가리키게 되면 한쪽의 쓰기가 다른 쪽 데이터를 덮어씁니다. 원인을 찾기 매우 어려운 전형적인 heap corruption 버그입니다.

셋째, Segmentation Fault 입니다. 잘못된 주소가 프로세스에 할당되지 않은 VA 영역이면 MMU 가 페이지 테이블을 뒤져도 PTE 가 invalid 로 남아 있습니다. MMU 는 예외를 올리고, 커널 폴트 핸들러는 "복구 불가능한 접근" 으로 판단해 프로세스에 SIGSEGV 를 보냅니다.

넷째, 보안 취약점입니다. 공격자가 힙 메타데이터를 의도적으로 조작하면 임의 주소에 쓰기(arbitrary write) 가 가능해집니다. 이는 흔히 코드 실행 공격의 시작점이 됩니다. 힙 익스플로잇 기법 대다수가 이 경로로 들어갑니다.

## 검증 — 세 가지 전형적인 실수

`malloc(100)` 으로 받은 블록의 실제 모양을 먼저 떠올려 두겠습니다.

```
 [ 헤더 (8B) ][ 사용자 데이터 (100B) ]
       ↑                 ↑
   블록 크기/할당비트   malloc 이 반환한 포인터 p

 free(p) 는 p 앞 8 바이트를 헤더로 읽어 "이 블록이 몇 바이트짜리구나" 를
 해석한 뒤 free list 에 넣습니다.
```

이 구조를 머리에 두면 아래 세 케이스가 왜 위험한지가 바로 보입니다.

### (a) double free

```c
char *p = malloc(100);
free(p);     // 1) 블록을 free list 에 삽입
free(p);     // 2) 같은 블록을 또 삽입
```

같은 블록이 free list 에 두 번 등록됩니다. 이후 `malloc` 을 두 번 호출하면 같은 주소가 두 번 반환됩니다.

```c
char *a = malloc(100);  // p 의 주소 반환
char *b = malloc(100);  // 또 p 의 주소 반환
```

`a` 와 `b` 가 같은 메모리를 가리키고, `a` 에 쓰면 `b` 가 덮어씌워집니다. 이 버그는 두 포인터의 조합 지점에서 터지므로 원인 추적이 매우 어렵습니다. 방어 관용구는 간단합니다. `free(p); p = NULL;` 을 함께 쓰면 두 번째 `free(NULL)` 은 표준에 의해 아무 일도 하지 않습니다.

### (b) free(p+1)

```c
char *p = malloc(100);
free(p + 1);  // 블록 시작이 아닌 1 바이트 뒤 주소
```

`free` 는 전달받은 주소 바로 앞 8 바이트를 헤더로 읽습니다. 그런데 `p+1` 바로 앞 8 바이트는 헤더가 아니라 사용자 데이터입니다.

```
 [ 헤더 ][ 데이터 ... ]
          ↑ p   ↑ p+1
               free 가 여기 앞 8B 를 헤더로 착각
```

사용자 데이터가 엉뚱한 값으로 헤더처럼 해석됩니다. "블록 크기 = 3 억 바이트" 같은 말도 안 되는 해석이 나올 수 있고, 그 해석으로 free list 를 조작하면 힙 전체가 오염됩니다. 바로 크래시하면 차라리 낫고, 조용히 데이터가 오염되는 쪽이 더 위험합니다.

### (c) int x; free(&x);

```c
int x = 42;
free(&x);  // 스택 변수의 주소를 free
```

`malloc` 은 힙 영역만 관리합니다. `&x` 는 스택 영역의 주소이므로 헤더가 있을 리 없습니다. 그 자리에는 다른 지역 변수나 함수 리턴 주소가 있습니다.

`free` 는 그 값을 헤더로 착각해 free list 에 등록합니다. 이후 `malloc` 은 스택 영역을 "빈 힙 블록" 으로 반환할 수 있고, 그 결과 스택과 힙이 뒤섞여 함수 리턴 주소가 덮어씌워질 수 있습니다. 공격자가 이 경로를 익스플로잇하면 원하는 코드를 실행하는 통로가 됩니다.

## 프로그래머의 규율

위 세 케이스가 주는 교훈은 같은 방향을 가리킵니다.

- `malloc` 이 반환한 원래 주소로만 `free` 합니다.
- `free` 후 해당 포인터를 `NULL` 로 설정합니다.
- 스택·전역 변수 주소는 절대 `free` 하지 않습니다.
- `free` 된 메모리를 다시 읽거나 쓰지 않습니다(use-after-free).

이 네 규칙을 지키지 않으면 힙 메타데이터가 조용히 무너지고, 증상은 훨씬 나중에 엉뚱한 곳에서 터집니다.

## 정리

`free()` 의 단순함은 게으름이 아니라 의도입니다. 검증을 포기해 성능을 얻고, 그 대가로 안전성을 호출자에게 넘겼습니다. 이 설계 결정은 C 의 다른 모든 부분과 일관됩니다. 배열 경계를 검사하지 않는 이유, 포인터 연산을 자유롭게 허용하는 이유도 같은 선택의 연장입니다. 빠른 `malloc`/`free` 의 대가가 메모리 버그의 구조적 위험이라는 사실은, 런타임을 가볍게 유지하기 위한 트레이드오프의 산물입니다.
