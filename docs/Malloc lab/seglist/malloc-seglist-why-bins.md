# Segregated Free List — 왜 크기별 bin 으로 나누는가

`malloc` 의 빠르기는 "빈 블록을 얼마나 빨리 찾느냐" 에서 결정됩니다. 가용 블록을 어떻게 조직하느냐가 탐색 비용을 직접 정하고, 그 조직 방식의 변화는 할당자 역사 자체를 이룹니다. Implicit → Explicit → Segregated 로 이어지는 발전은 결국 탐색 복잡도를 줄이는 과정입니다.

## 세 가지 자료구조의 탐색 비용

세 구조를 한 줄씩 비교하면 차이가 선명합니다.

- Implicit Free List — 모든 블록(할당 + 가용) 을 순차 스캔합니다. 할당 블록도 건너뛰며 봐야 하므로 복잡도가 O(전체 블록 수) 입니다.
- Explicit Free List — 가용 블록만 이중 연결 리스트로 엮습니다. 복잡도가 O(가용 블록 수) 로 줄지만, 여전히 가용 블록이 많으면 느립니다.
- Segregated Free List — 가용 블록을 크기별로 나눈 여러 리스트(bin) 에 담습니다. `malloc(n)` 은 n 에 해당하는 bin 만 보면 되므로 탐색 범위가 크게 줄고, 잘 설계하면 거의 O(1) 에 가까워집니다.

```
 Implicit  : 전체 블록 순회      O(전체 블록 수)
 Explicit  : 가용 블록만 순회    O(가용 블록 수)
 Segregated: 해당 bin 만 탐색    ≈ O(1)
```

세 단계는 "탐색 대상 집합을 좁혀 가는" 한 방향의 진화입니다.

## bin 을 나누는 기준

bin 설계의 원칙은 요청 분포에 맞추는 것입니다. 현실의 `malloc` 요청은 작은 크기에 압도적으로 몰리고, 큰 크기는 드물지만 편차가 큽니다. 그래서 작은 쪽은 정확하게, 큰 쪽은 범위를 넓게 잡습니다.

- 작은 크기 — 8B 단위로 정확히 끊습니다(8, 16, 24, 32, ...). 작은 할당이 빈번하니 내부 단편화를 줄이는 것이 이득입니다.
- 중간 크기 — 2 의 거듭제곱 구간으로 묶습니다(33~64, 65~128, 129~256, ...). bin 개수를 합리적으로 유지하기 위해서입니다.
- 큰 크기 — 단일 bin 에 모두 넣거나 크기 순으로 정렬해 관리합니다. 드문 요청에 대한 비용은 중요하지 않습니다.

glibc 의 ptmalloc 은 이 원칙을 그대로 따라 fastbin(작은 크기, LIFO), smallbin, largebin 의 3 단계 구조를 씁니다. 크기 계층이 실제 요청 분포에 맞춰져 있어 작은 요청은 상수 시간에 가깝게 해결됩니다.

## 검증 — malloc(12) 이 bin 3 개짜리 시스템에서 흐르는 경로

bin 구성과 요청을 구체적으로 두고 따라가 보겠습니다.

```
  bin[0] : 1~8B   가용 블록들
  bin[1] : 9~16B  가용 블록들
  bin[2] : 17~32B 가용 블록들

  요청: malloc(12)
```

처리 흐름은 다음과 같습니다.

1. 실제 필요 크기를 결정합니다. 헤더 8B 와 정렬 패딩을 고려해 대략 16B~24B 가 필요합니다(여기서는 요청 12B 기준으로 bin 을 선택).
2. 12B 는 `9~16B` 구간에 해당하므로 `bin[1]` 을 탐색합니다.
3. `bin[1]` 에 가용 블록이 있으면 그 블록을 할당하고 리스트에서 제거합니다. 없으면 더 큰 `bin[2]` 에서 32B 블록을 꺼내 16B 로 split 한 뒤 나머지 16B 를 `bin[1]` 에 삽입합니다.
4. `bin[2]` 도 비어 있으면 `sbrk`/`mmap` 으로 OS 에 새 메모리를 요청합니다.
5. 최종 블록의 헤더에 할당 비트를 설정하고, 헤더 다음 주소를 사용자에게 반환합니다.

내부 단편화는 여기서도 얹힙니다. 12B 를 요청했는데 16B 블록이 할당되면 4B 가 패딩으로 낭비됩니다. bin 을 더 촘촘히 나누면 이 낭비가 줄지만 bin 수 자체가 많아지면 관리 오버헤드가 커집니다. 실무 할당기의 bin 구성은 이 두 비용을 타협한 결과입니다.

## 배치 정책과의 결합

segregated list 의 "해당 bin 안에서 아무거나" 선택은 사실상 best-fit 근사를 상수 시간에 달성합니다. bin 자체가 이미 크기 구간으로 나뉘어 있으므로, 그 안에서 첫 블록을 집어도 요청 크기와 충분히 가깝습니다. 그래서 현대 할당기는 대개 segregated 구조 + class 내 first-fit 조합을 기본으로 씁니다.

## 정리

Segregated Free List 의 핵심 아이디어는 "탐색 비용은 리스트 크기에 비례하므로, 리스트를 잘게 나눠 탐색 집합 자체를 줄이자" 입니다. 작은 할당에 bin 을 촘촘히 두어 내부 단편화를 줄이고, 큰 할당에는 관리 부담을 아낍니다. 이 설계는 O(1) 에 가까운 처리량과 best-fit 근사의 단편화 억제를 한 자료구조 안에서 얻어냅니다. 할당자 내부 자료구조 중에서 가장 구조적으로 잘 정리된 것이 바로 이 bin 구조입니다.
