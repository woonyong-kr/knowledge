# 배치 정책 — First Fit·Next Fit·Best Fit 의 실전 트레이드오프

가용 블록이 여러 개라면 할당기는 그중 어떤 것을 선택할지 결정해야 합니다. 이 선택 방식이 배치 정책(placement policy) 입니다. 단순해 보이지만 정책이 단편화 경향과 탐색 비용을 모두 바꾸므로, 할당자 성능에 직접 영향을 주는 결정입니다.

## 세 정책의 탐색 비용과 단편화 경향

### First Fit

free list 의 처음부터 탐색해 "들어가는" 첫 블록을 선택합니다. 최악 복잡도는 O(가용 블록 수) 이지만 평균적으로는 빠른 편입니다. 시간이 지나면 리스트 앞쪽이 반복 split 되면서 작은 조각이 앞쪽에 누적되는 경향이 생깁니다. 결과적으로 뒤쪽에는 큰 블록이 남아 있고, 앞쪽 탐색은 점점 느려집니다.

### Next Fit

마지막으로 탐색을 멈춘 위치부터 이어서 탐색합니다. First Fit 의 앞쪽 편향을 피하려는 변형입니다. 평균 처리량은 더 빠를 수 있지만 조각이 리스트 전체에 고르게 퍼지고, 큰 블록도 쉽게 쪼개져서 외부 단편화가 실제로 First Fit 보다 악화되는 경우가 보고됩니다. 메모리 이용률 면에서는 일반적으로 불리합니다.

### Best Fit

전체 free list 를 탐색해 요청 크기에 가장 가까운 블록을 선택합니다. split 잔여를 최소화하므로 내부 단편화가 가장 적고 메모리 이용률이 가장 좋습니다. 대신 매 요청마다 리스트 전체를 훑어야 하므로 탐색 비용이 가장 큽니다. 다만 segregated list 에서는 bin 자체가 이미 크기로 나뉘어 있어 bin 안에서 first-fit 만 해도 best-fit 근사를 얻고, 탐색은 거의 O(1) 이 됩니다.

### 비교 요약

```
 정책       | 탐색 속도 | 단편화 경향     | 실전 평가
 -----------|----------|------------------|-----------------------------
 First Fit  | 중간      | 앞쪽에 조각 집중 | 단순하고 무난한 기본 선택
 Next Fit   | 빠름      | 전체에 고르게 나쁨| 큰 블록 보존에 불리
 Best Fit   | 느림      | 잔여 최소        | segregated list 와 결합 시 최적
```

## 검증 — 같은 가용 리스트에서 세 정책이 어떻게 다르게 동작하는가

구체 수치로 비교하면 정책 차이가 선명합니다.

```
 전제 : 최소 블록 크기 16B (헤더 8B + 최소 페이로드 8B)
 요청 : malloc(20) → 정렬 후 필요한 블록 크기 24B
 가용 리스트 : [32B] → [64B] → [24B] → [128B]
```

### First Fit

리스트 앞에서부터 탐색하면 32B 블록이 첫 적합이 됩니다. 잔여는 `32 - 24 = 8B` 이고, 8B 는 최소 블록 크기 16B 미만이므로 split 이 불가능합니다. 결과적으로 32B 전체가 할당되고 8B 내부 단편화가 발생합니다.

### Next Fit

이전 탐색 위치가 리스트 시작이었다고 가정하면 First Fit 과 동일한 결과입니다. 만약 이전 위치가 64B 이후였다면 24B 블록에 도달해 잔여 0B 로 딱 맞게 할당합니다. 결과가 탐색 시작 위치에 따라 달라지는 것이 Next Fit 의 특성입니다.

### Best Fit

리스트 전체를 훑어 요청 24B 에 가장 가까운 블록을 고릅니다.

```
 32B  → 잔여 8B
 64B  → 잔여 40B
 24B  → 잔여 0B   ← 가장 가까움
 128B → 잔여 104B
```

24B 블록이 선택되고 잔여는 0B 입니다. split 필요 없이 완벽한 매칭이 이루어지고 단편화가 없습니다.

### 결과 비교

```
 정책       | 선택 블록 | 잔여   | split | 단편화
 -----------|----------|--------|-------|--------
 First Fit  | 32B      | 8B     | 불가  | 내부 8B
 Next Fit   | 32B 또는 24B | 경우 따라 다름 | — | —
 Best Fit   | 24B      | 0B     | 없음  | 없음
```

이 한 케이스에서는 Best Fit 이 명백히 유리합니다. 다만 Best Fit 도 항상 최적은 아닙니다. 작은 잔여 조각이 반복해서 생기면 그것들이 다시 외부 단편화로 이어질 수 있고, 전체 리스트 순회 비용이 처리량을 갉아먹습니다. 그래서 실무 할당기는 Best Fit 단독 대신 "segregated + 클래스 내 first-fit" 조합으로 Best Fit 의 근사를 상수 시간에 얻는 방향을 택합니다.

## 정리

배치 정책 자체에 절대 강자는 없습니다. First Fit 은 단순함과 평균 속도, Next Fit 은 앞쪽 편향 회피, Best Fit 은 이용률이 강점이고 각각 서로의 약점을 갖고 있습니다. 그래서 현대 할당기는 자료구조(segregated list) 와 정책(bin 내 first-fit) 을 결합해 세 정책의 장점을 동시에 추구합니다. "어떤 정책을 쓰느냐" 는 "어떤 자료구조와 함께 쓰느냐" 와 떼어 놓고 판단할 수 없는 문제입니다.
