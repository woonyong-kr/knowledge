# 내부 단편화 vs 외부 단편화 — 숫자로 체감하는 두 종류의 낭비

동적 메모리 할당자를 공부하다 보면 "힙에 빈 공간이 있는데 왜 `malloc` 이 실패하는가" 라는 순간을 만납니다. 이 질문의 답은 단편화(fragmentation) 에 있습니다. 단편화는 한 종류가 아니라 두 종류이고, 둘의 원인과 완화 기법도 다릅니다.

## 내부 단편화 vs 외부 단편화

내부 단편화(Internal Fragmentation) 는 할당된 블록 내부에서 실제로 쓰지 않는 부분이 낭비되는 경우입니다. 예컨대 13 바이트를 요청했는데 정렬 때문에 16 바이트 블록이 할당되면 3 바이트가 패딩으로 낭비됩니다. 블록 "안쪽" 의 낭비라서 내부 단편화라고 부릅니다.

외부 단편화(External Fragmentation) 는 빈 공간의 총합은 충분하지만 연속되지 않아 큰 블록을 할당할 수 없는 상태입니다. 48 B 짜리 가용 공간과 32 B 짜리 가용 공간이 따로 있으면 합치면 80 B 지만 60 B 짜리 연속 블록은 만들 수 없습니다. 블록 "바깥" 에서 조각난 공간이 문제라서 외부 단편화라고 부릅니다.

## 각각을 줄이는 전략

내부 단편화를 줄이는 쪽은 비교적 단순합니다.

- 정렬 단위를 줄이면 패딩이 줄어듭니다(대신 성능 트레이드오프가 따라옵니다).
- 요청 크기에 맞는 size class 를 쓰는 Segregated List 구조를 채택합니다.
- 작은 블록 전용 할당기(slab allocator 등) 를 별도로 운영합니다.

외부 단편화는 구조적으로 완전히 없앨 수는 없고 다음 기법으로 완화합니다.

- Coalescing — 인접한 빈 블록을 합쳐 큰 블록으로 만듭니다.
- Best Fit — 요청 크기에 가장 가까운 빈 블록을 선택해 잔여 조각을 최소화합니다.
- Compaction — 사용 중인 블록을 한쪽으로 모아 빈 공간을 연속으로 만듭니다. 다만 GC 언어에서만 가능하고, C 의 `malloc` 은 포인터가 바뀌므로 불가능합니다.

## 검증 — 왜 malloc(60) 이 실패하는가

다음 힙 상태를 상정합니다.

```
[16B 할당][48B 가용][16B 할당][32B 가용]

총 가용 공간 = 48 + 32 = 80B
요청 크기    = 60B
```

80 B 의 가용 공간이 있지만 연속된 60 B 블록은 없습니다.

- 48 B 블록 → 60 B 요청에 12 B 부족.
- 32 B 블록 → 60 B 요청에 28 B 부족.
- 두 가용 블록 사이에 16 B 할당 블록이 끼어 있어 합칠 수 없습니다.

총량은 충분하지만 조각나 있어서 할당이 불가능한 이 상태가 외부 단편화의 전형입니다.

실제 할당에는 헤더와 정렬 패딩까지 포함됩니다. `malloc(60)` 의 실제 블록 크기는 헤더 8 B 를 더해 68 B, 정렬로 올림하면 72 B 또는 80 B 가 될 수 있습니다. 48 B 가용 블록은 72 B 요청에 더더욱 부족합니다. 내부 단편화가 외부 단편화 위에 얹히는 구도입니다.

해결은 coalescing 으로 가능합니다. 중간의 16 B 할당 블록이 free 되면 48 + 16 + 32 = 96 B 연속 블록을 확보할 수 있고, 배치 정책을 잘 잡았다면 조각이 덜 난 상태로 유지할 수 있습니다.

## 정리

단편화는 "메모리가 모자란 것" 이 아니라 "쓸 수 있는 모양으로 남아 있지 않은 것" 에서 생깁니다. 내부 단편화는 정렬과 최소 블록 규약의 부산물, 외부 단편화는 시간에 따른 할당·해제 패턴의 부산물입니다. 둘의 원인이 다르니 전략도 다르고, 할당기 설계가 두 종류의 단편화를 각각 어떻게 제어하는지가 성능을 가릅니다.
