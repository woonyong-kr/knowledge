## 1. 왜 페이지를 노드로 사용하는가

### 교과서의 B+ tree 와 minidb 의 B+ tree

교과서에서 B+ tree 는 보통 이렇게 표현됩니다.

```
교과서:
  노드 = 메모리의 객체 (malloc 된 구조체)
  자식 연결 = 포인터 (메모리 주소)
  트리 전체가 메모리에 상주

  struct node {
      int keys[ORDER];
      struct node *children[ORDER+1];  ← 메모리 포인터
  };
```

하지만 데이터베이스는 디스크 기반입니다. 메모리 포인터는 프로그램이 종료되면 사라집니다.

```
minidb:
  노드 = 디스크의 페이지 (4096 바이트 블록)
  자식 연결 = page_id (파일 내 위치를 나타내는 정수)
  필요한 노드만 디스크에서 읽어옴

  페이지 안에:
  [page_type][parent_page_id][key_count][leftmost_child_page_id]
  [key=146, right_child_page_id=4]
  [key=292, right_child_page_id=6]
  ...
```

### 페이지와 노드 설계의 이점

디스크에서 데이터를 읽을 때 최소 단위는 OS 페이지(4096 바이트) 입니다. 노드 하나가 100 바이트여도 디스크에서는 4096 바이트를 읽어야 합니다. 그렇다면 4096 바이트를 통째로 하나의 노드로 쓰는 편이 효율적입니다.

```
노드가 작을 때 (100 바이트 노드):
  1 회 디스크 읽기 = 4096 바이트 → 노드 1 개만 사용 → 3996 바이트 낭비
  트리 높이: 깊어짐 → 더 많은 디스크 읽기 필요

페이지 = 노드 (4096 바이트 노드):
  1 회 디스크 읽기 = 4096 바이트 → 노드 1 개 전체 활용 → 낭비 없음
  내부 노드 하나에 key 340 개 → 자식 341 개를 안내
  → 트리가 매우 납작해짐 → 디스크 읽기 횟수 최소화
```

```
이진 트리(key 2개/노드) 로 1000 만 건 검색:
  높이 = log₂(10,000,000) ≈ 24 단
  → 24 번 디스크 읽기

B+ tree(key 340개/노드) 로 1000 만 건 검색:
  높이 = log₃₄₁(10,000,000) ≈ 3 단
  → 3 번 디스크 읽기

디스크 접근 8 배 감소
```

OS 커널도 파일 I/O 를 페이지 단위로 캐싱합니다. DB 페이지 크기가 OS 페이지 크기와 일치하면 하나의 DB 페이지가 정확히 하나의 OS 캐시 라인에 대응하여 효율적입니다.

## 2. 초기 상태 — 빈 트리

`pager_open(create=true)` 실행 직후 상태는 다음과 같습니다.

```
page 2 (리프, 루트):
  ┌─────────────────────────────────────────────┐
  │ type=LEAF, parent=0, key_count=0            │
  │ next_leaf=0, prev_leaf=0                    │
  │                                             │
  │ (엔트리 없음 — 빈 리프)                      │
  └─────────────────────────────────────────────┘

헤더: root_index_page_id = 2
트리 높이: 1 (리프만 있음)
```

## 3. INSERT 과정 — 리프에 공간이 있을 때

```
INSERT id=1 → INSERT id=2 → INSERT id=3

page 2 (리프, 루트):
  ┌────────────────────────────────────────────────────────────┐
  │ type=LEAF, parent=0, key_count=3                          │
  │ [key=1, ref={p1,s0}] [key=2, ref={p1,s1}] [key=3, ref={p1,s2}] │
  └────────────────────────────────────────────────────────────┘
```

정렬 삽입 과정 (key=2 삽입 시 기존 key=1, key=3 가 있다고 가정):

```
기존:  [key=1] [key=3]
삽입:  key=2, 위치=1 (이진 탐색으로 결정)

임시 배열(tmp) 에 조립:
  (1) memcpy: tmp[0] = entries[0]        → [key=1]
  (2) 직접:   tmp[1] = {key=2, ref}      → [key=1][key=2]
  (3) memcpy: tmp[2] = entries[1]        → [key=1][key=2][key=3]

tmp 를 페이지에 다시 기록
```

## 4. 리프 분할 — 291 개 초과 시

```
page 2 (리프, 291 개 꽉 참):
  [key=1] [key=2] [key=3] ... [key=145] [key=146] ... [key=291]

key=292 삽입 시도 → 공간 없음 → 분할 발생
```

STEP 1. 임시 배열에 292 개 조립

```
tmp[292]:
  [key=1] [key=2] ... [key=145] | [key=146] [key=147] ... [key=291] [key=292]
                                ↑
                           split = 146 (절반 지점)
```

STEP 2. 앞쪽 절반은 기존 페이지에

```
page 2 (리프):
  key_count = 145
  [key=1] [key=2] ... [key=145]
```

STEP 3. 뒤쪽 절반은 새 페이지에

```
page 4 (새 리프, pager_alloc_page() 로 할당):
  key_count = 147
  [key=146] [key=147] ... [key=292]
```

STEP 4. 형제 연결 갱신

```
page 2: next_leaf = page 4
page 4: prev_leaf = page 2, next_leaf = 0
```

STEP 5. 승격 — 부모에 구분 key 전달

```
promote_key = 146 (뒤쪽 페이지의 첫 번째 key)
→ "146 이상은 page 4 로 가라" 는 정보를 부모에 삽입
```

## 5. 새 루트 생성 — 최초 분할 시

page 2 가 루트였으므로 부모가 없습니다. 새 루트를 만듭니다.

```
분할 전:
  root = page 2 (리프)

분할 후:
  page 5 (새 내부 노드, 새 루트):
    ┌─────────────────────────────────────────────────┐
    │ type=INTERNAL, parent=0, key_count=1            │
    │ leftmost_child = page 2                         │
    │ [key=146, right_child=page 4]                   │
    └─────────────────────────────────────────────────┘
    "146 미만 → page 2, 146 이상 → page 4"

  page 2 (리프): parent = page 5, keys 1~145
  page 4 (리프): parent = page 5, keys 146~292

헤더: root_index_page_id = 2 → 5 로 변경
트리 높이: 1 → 2
```

```
       [page 5: 내부 노드]
        key=146
       /        \
[page 2: 리프]  [page 4: 리프]
 key 1~145       key 146~292
```

## 6. 연쇄 분할 — 계속 INSERT 시

INSERT 가 계속되어 리프 4 개로 성장한 트리는 다음과 같이 생겼습니다.

```
              [page 5: 내부 노드]
         key=146, key=292, key=438
        /       |        |        \
  [page 2]  [page 4]  [page 6]  [page 8]
  key 1~145  key 146~291 key 292~437 key 438~584
```

page 5 의 내부 노드도 최대 340 개 key 까지 저장 가능합니다. 341 번째 key 가 승격되면 page 5 도 분할됩니다.

```
STEP 1: page 5 분할
  page 5:  앞쪽 170 개 key
  page 10: 뒤쪽 169 개 key (새 페이지)
  promote_key = 중간값

STEP 2: 새 루트 생성
  page 11 (새 루트):
    leftmost_child = page 5
    [key=중간값, right_child=page 10]

헤더: root_index_page_id = 5 → 11 로 변경
트리 높이: 2 → 3
```

## 7. 삭제 과정

```
DELETE key=300 (page 6 리프에 있다고 가정):

page 6 리프:
  삭제 전: [key=292] [key=293] ... [key=300] [key=301] ... [key=437]
                                    ↑ 이걸 제거

  (1) key=300 의 위치(idx) 찾기 (이진 탐색)
  (2) idx 뒤의 엔트리를 한 칸씩 앞으로 당김 (memcpy)
  (3) key_count-- (146 → 145)

  삭제 후: [key=292] [key=293] ... [key=299] [key=301] ... [key=437]
```

minidb 는 단순 구현으로, 리프가 비어도 노드 병합(merge) 은 하지 않습니다. 리프의 모든 key 가 삭제되면 해당 페이지를 free 리스트에 반환합니다.

## 9. B+ tree 의 핵심 불변식

```
1. 모든 데이터(row_ref) 는 리프에만 존재합니다 (내부 노드에는 없음)
2. 리프는 key 순서로 정렬되어 있습니다
3. 리프끼리 이중 연결 리스트로 연결되어 있습니다
4. 내부 노드의 key 는 자식을 분류하는 "구분자" 일 뿐입니다
5. 루트를 제외한 모든 노드는 최소 절반 이상 채워져 있습니다
6. 트리는 항상 균형입니다 (모든 리프의 깊이가 동일)
7. 트리는 아래에서 위로 자랍니다 (분할 → 승격)
```

이 일곱 가지 불변식이 "디스크 페이지 위의 B+ tree" 를 지탱합니다. 각 불변식을 깨뜨리지 않는 연산 순서만 지키면 인덱스가 바르게 유지됩니다.
