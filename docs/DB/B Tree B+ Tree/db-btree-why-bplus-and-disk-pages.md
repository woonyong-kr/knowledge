## 0. 이 문서의 목적

이 문서는 다음 질문에 답하기 위해 만듭니다.

- 메모리 기반 B+ Tree 와 디스크 기반 B+ Tree 는 무엇이 다른가?
- B+ Tree 가 디스크 페이지와 어떻게 연결되는가?
- 왜 이 조합이 좋은가?
- 실제 DBMS 는 이 둘을 어떻게 쓰는가?

이 개념을 이해하지 않으면, B+ Tree 를 구현해도 "왜 이걸 쓰는지" 설명할 수 없습니다.

## 1. 먼저, 문제를 정의합니다

### 1.1 데이터가 100 만 건이면 무슨 일이 생기는가

row 하나가 128 바이트라고 가정합니다.

```
100 만 건 × 128 바이트 = 약 128 MB
```

이걸 전부 메모리에 올리면 지금은 괜찮을 수 있습니다. 하지만 1000 만 건이면 1.28 GB, 1 억 건이면 12.8 GB 입니다. 실제 DB 는 메모리보다 데이터가 훨씬 큽니다.

그래서 DB 는 데이터를 파일에 저장하고, 필요한 부분만 메모리에 올려서 씁니다.

### 1.2 파일에서 데이터를 찾는 비용

파일에서 특정 row 를 찾으려면 파일을 읽어야 합니다. 디스크 읽기는 메모리 접근보다 수만에서 수십만 배 느립니다.

```
메모리 접근: ~100ns (나노초)
SSD 랜덤 읽기: ~100,000ns (0.1ms)
HDD 랜덤 읽기: ~10,000,000ns (10ms)
```

따라서 DB 의 성능은 파일을 몇 번 읽느냐로 결정됩니다. CPU 연산 횟수가 아닙니다.

## 2. 파일을 페이지 단위로 다루는 이유

### 2.1 OS 가 이미 페이지 단위로 일합니다

CSAPP 9 장에서 배운 것처럼, OS 는 메모리를 4 KB 페이지 단위로 관리합니다. 파일을 읽을 때도 커널은 페이지 캐시를 통해 4 KB 단위로 읽고 씁니다. 1 바이트를 읽으라고 해도 OS 는 내부적으로 4 KB 를 읽습니다. 그렇다면 DB 도 처음부터 4 KB 단위로 읽고 쓰는 게 자연스럽습니다.

### 2.2 페이지 = 파일의 최소 읽기 단위

DB 에서 페이지란 다음을 의미합니다.

```
database.db 파일을 4096 바이트씩 자른 조각
```

```
page 0: 오프셋 0 ~ 4095
page 1: 오프셋 4096 ~ 8191
page 2: 오프셋 8192 ~ 12287
...
page N: 오프셋 N*4096 ~ (N+1)*4096-1
```

특정 페이지를 읽으려면 다음 한 줄이면 충분합니다.

```c
pread(fd, buffer, 4096, page_id * 4096);
```

파일 전체를 읽을 필요가 없습니다. 이것이 페이지 단위 저장의 가장 근본적인 이점입니다.

## 3. 메모리 B+ Tree 와 디스크 B+ Tree 의 차이

### 3.1 메모리 기반 B+ Tree

노드를 `malloc` 으로 할당하고, 자식을 메모리 포인터로 가리킵니다.

```c
struct Node {
    int keys[ORDER];
    struct Node *children[ORDER + 1];
};
```

fan-out 이 2 이든 4 이든 성능 차이는 작습니다. 모든 것이 메모리에 있기 때문입니다. 트리 높이가 log₂N 이든 log₁₆N 이든 각 노드 접근은 수십 나노초 수준이라 총 시간이 비슷합니다.

### 3.2 디스크 기반 B+ Tree

노드를 페이지에 박고, 자식을 page_id 로 가리킵니다.

```c
typedef struct __attribute__((packed)) {
    uint32_t page_type;      // PAGE_TYPE_INTERNAL
    uint32_t parent_page_id;
    uint32_t key_count;
    uint32_t leftmost_child_page_id;
    // key_count 개의 {key, right_child_page_id}
} internal_page_header_t;
```

이 환경에서는 fan-out 이 곧 트리 높이를 결정합니다. 트리 높이가 곧 디스크 I/O 횟수를 결정합니다.

## 4. fan-out 이 커야 하는 이유

한 페이지에 몇 개의 키를 넣을 수 있는지가 fan-out 을 결정합니다. minidb 기준:

```
내부 노드 페이지 (4096 B):
  헤더: 16 B
  엔트리: 12 B (key 8B + child_page_id 4B)
  최대 엔트리: (4096 - 16) / 12 = 340
  fan-out: 340 + 1 = 341 (leftmost child 포함)
```

fan-out 이 341 이면 트리 높이는 다음과 같이 예측됩니다.

```
높이 1: 최대 341 개 키
높이 2: 최대 341 × 341 ≈ 116,281 개 키
높이 3: 최대 341³ ≈ 3,965 만 개 키
높이 4: 최대 341⁴ ≈ 135 억 개 키
```

1 억 건 데이터라도 디스크 I/O 4 번이면 리프에 도달합니다. 바이너리 트리 라면 높이가 27 이 되어 27 번의 I/O 가 필요합니다. 이 차이가 "왜 B+ tree 인가" 에 대한 답의 전부입니다.

## 5. 한 번의 디스크 I/O 를 낭비하지 않는 설계

페이지 크기를 OS 와 디스크 단위와 같게 맞추면, "한 페이지 읽기 = 한 I/O" 가 성립합니다. 페이지를 작게 쪼개면 I/O 당 얻는 정보가 줄고, 크게 늘리면 한 I/O 의 비용이 올라갑니다. 4 KB 에서 균형이 잡힙니다.

한 I/O 로 얻은 4 KB 안에 fan-out 이 높은 노드 하나가 들어가야 "그 I/O 가 값어치를 한다" 라고 말할 수 있습니다. 그래서 B+ tree 노드 = 한 페이지 = 4 KB 라는 세 단위의 정렬이 자연스럽습니다.

## 6. 실제 DBMS 의 선택

SQLite, PostgreSQL, MySQL InnoDB 가 모두 B+ tree 혹은 그 변형(B-tree 클러스터드 인덱스) 을 씁니다. 페이지 크기만 조금씩 다를 뿐입니다.

- SQLite: 기본 4 KB (1 KB ~ 64 KB 구성 가능)
- PostgreSQL: 8 KB 기본
- MySQL InnoDB: 16 KB 기본

공통 원리는 같습니다. "디스크 I/O 가 병목이면, 한 I/O 로 최대한 많은 정보를 가져온다. 그 정보가 다음 I/O 의 방향을 결정하는 인덱스라면 더 좋다." B+ tree 는 이 원리를 가장 깔끔하게 구현한 자료구조이고, 그 깔끔함을 성립시키는 조건이 바로 페이지 기반 저장입니다.

## 7. 정리

메모리 B+ tree 에서는 fan-out 이 선택 사항이지만, 디스크 B+ tree 에서는 fan-out 이 설계의 중심입니다. 한 페이지가 한 노드가 되고, 한 노드가 한 I/O 를 담당하므로, "한 I/O 에 몇 개의 키를 싣느냐" 가 전체 성능을 결정합니다. 이 단순한 규칙이 디스크 기반 DB 엔진의 골격을 지탱합니다.
