## 개요

minidb 의 모든 데이터는 고정 크기 페이지(4096 바이트) 단위로 디스크에 저장됩니다. 하나의 DB 파일은 연속된 페이지의 배열이며, 각 페이지는 첫 4 바이트(`page_type`) 로 용도가 결정됩니다.

```
DB 파일 (sql.db):
┌──────────┬──────────┬──────────┬──────────┬──────────┬──────────┐
│  page 0  │  page 1  │  page 2  │  page 3  │  page 4  │  page 5  │
│  HEADER  │   HEAP   │   LEAF   │   HEAP   │   LEAF   │ INTERNAL │
│ 4096 B   │ 4096 B   │ 4096 B   │ 4096 B   │ 4096 B   │ 4096 B   │
└──────────┴──────────┴──────────┴──────────┴──────────┴──────────┘
  0x0000     0x1000     0x2000     0x3000     0x4000     0x5000
```

페이지 크기는 `sysconf(_SC_PAGESIZE)` 로 OS 에서 결정됩니다.

- x86_64 리눅스: 4096 바이트 (4 KB)
- Apple Silicon macOS: 16384 바이트 (16 KB)

## 페이지 타입 코드

| 코드   | 상수                 | 용도                       |
| ------ | -------------------- | -------------------------- |
| `0x01` | `PAGE_TYPE_HEADER`   | DB 헤더 (page 0 전용)      |
| `0x02` | `PAGE_TYPE_HEAP`     | 힙 데이터 페이지 (행 저장) |
| `0x03` | `PAGE_TYPE_LEAF`     | B+ tree 리프 노드          |
| `0x04` | `PAGE_TYPE_INTERNAL` | B+ tree 내부 노드          |
| `0x05` | `PAGE_TYPE_FREE`     | 빈 페이지 (재활용 대기)    |

## 1. DB 헤더 페이지 (page 0)

데이터베이스의 전역 메타데이터를 저장합니다. 프로그램 시작 시 이 페이지 하나만 읽으면 DB 전체 상태를 파악할 수 있습니다.

### 메모리 레이아웃 (676 바이트, 나머지는 0 으로 패딩)

```
오프셋  크기   필드                    예시 값
──────────────────────────────────────────────────────────────
0x00    8B     magic                   "MINIDB\0\0"
0x08    4B     version                 1
0x0C    4B     page_size               4096
0x10    4B     root_index_page_id      2       ← B+ tree 루트 위치
0x14    4B     first_heap_page_id      1       ← 힙 체인 시작
0x18    4B     next_page_id            6       ← 다음 할당할 페이지 번호
0x1C    4B     free_page_head          0       ← 빈 페이지 리스트 (0=없음)
0x20    8B     next_id                 101     ← 다음 자동 증가 ID
0x28    8B     row_count               100     ← 살아있는 행 수
0x30    2B     column_count            3       ← 컬럼 수
0x32    2B     row_size                44      ← 행 직렬화 크기
0x34    624B   columns[16]             컬럼 메타데이터 배열
──────────────────────────────────────────────────────────────
합계: 676 바이트 (4096 바이트 페이지의 16.5%)
```

### columns[] 배열 — 스키마 정의

`CREATE TABLE users (name VARCHAR(32), age INT)` 실행 후 컬럼 메타는 다음과 같이 배치됩니다.

```
columns[0]: name="id"    type=BIGINT(2)  size=8   offset=0   is_system=1
columns[1]: name="name"  type=VARCHAR(3) size=32  offset=8   is_system=0
columns[2]: name="age"   type=INT(1)     size=4   offset=40  is_system=0

→ row_size = 8 + 32 + 4 = 44 바이트
```

## 2. 힙 페이지 (PAGE_TYPE_HEAP)

실제 행 데이터를 저장하는 페이지입니다. 슬롯 디렉터리는 앞에서 뒤로, 행 데이터는 뒤에서 앞으로 성장합니다.

```
오프셋 0                                                        오프셋 4096
┌────────┬────────┬────────┬────────┬────────────────┬──────┬──────┬──────┐
│ 헤더   │ slot 0 │ slot 1 │ slot 2 │    빈 공간     │ row2 │ row1 │ row0 │
│  16B   │   8B   │   8B   │   8B   │    3884B       │  44B │  44B │  44B │
└────────┴────────┴────────┴────────┴────────────────┴──────┴──────┴──────┘
|←──── 앞에서 뒤로 성장 ────→|                        |←── 뒤에서 앞으로 성장 ──→|
```

### 힙 페이지 헤더 (16 바이트)

```
오프셋  크기   필드                 설명
──────────────────────────────────────────────────────────
0x00    4B     page_type            PAGE_TYPE_HEAP (0x02)
0x04    4B     next_heap_page_id    다음 힙 페이지 (연결 리스트, 0=끝)
0x08    2B     slot_count           할당된 슬롯 총 수 (빈 슬롯 포함)
0x0A    2B     free_slot_head       빈 슬롯 체인 헤드 (0xFFFF=없음)
0x0C    2B     free_space_offset    뒤에서부터 사용된 행 데이터 크기
0x0E    2B     reserved             예약 (정렬용)
──────────────────────────────────────────────────────────
합계: 16 바이트
```

### 슬롯 엔트리 (8 바이트)

```
오프셋  크기   필드        설명
────────────────────────────────────────────
0x00    2B     offset      페이지 내 행 데이터 시작 위치
0x02    2B     status      ALIVE(0x01) / FREE(0x03)
0x04    2B     next_free   빈 슬롯 체인의 다음 (FREE 일 때만)
0x06    2B     reserved    예약
────────────────────────────────────────────
합계: 8 바이트
```

### 행 데이터 저장 예시 (row_size=44)

```
INSERT INTO users VALUES ('Alice', 25)   → id=1 자동 할당

행 직렬화 (44 바이트):
오프셋  크기   값
──────────────────────────────────────────
0x00    8B     01 00 00 00 00 00 00 00    ← id=1 (BIGINT)
0x08    32B    41 6C 69 63 65 00 ... 00   ← "Alice" + 패딩 (VARCHAR)
0x28    4B     19 00 00 00                ← age=25 (INT)
──────────────────────────────────────────

이 44 바이트가 page 1 의 오프셋 4052(= 4096 - 44) 에 저장됩니다.
slot 0 = { offset=4052, status=ALIVE }
```

### 공간 계산

```
slots_end       = 헤더(16) + 슬롯수 × 8
data_start      = page_size - free_space_offset
사용 가능 공간   = data_start - slots_end
페이지당 최대 행 = (4096 - 16) / (44 + 8) = 4080 / 52 ≈ 78 개
```

### 삭제와 재활용 (톰스톤 방식)

```
DELETE WHERE id=2 (slot 1 삭제):

  삭제 전:                           삭제 후:
  slot 0: ALIVE, offset=4052        slot 0: ALIVE, offset=4052
  slot 1: ALIVE, offset=4008        slot 1: FREE,  next_free=NONE ← 톰스톤
  slot 2: ALIVE, offset=3964        slot 2: ALIVE, offset=3964
  free_slot_head = NONE              free_slot_head = 1

다음 INSERT 시:
  free_slot_head=1 → slot 1 재활용
  offset=4008 위치에 새 데이터 덮어씀
  slot 1: ALIVE, offset=4008 (재사용)
  free_slot_head = NONE
```

## 3. B+ tree 리프 페이지 (PAGE_TYPE_LEAF)

key → 힙 위치 매핑(인덱스) 을 저장합니다. 엔트리는 key 순서로 정렬되어 있습니다.

```
오프셋 0                                                     오프셋 4096
┌────────────┬──────────┬──────────┬──────────┬──────────────────────┐
│  리프 헤더 │ entry 0  │ entry 1  │ entry 2  │      빈 공간         │
│    20B     │   14B    │   14B    │   14B    │                      │
└────────────┴──────────┴──────────┴──────────┴──────────────────────┘
```

### 리프 헤더 (20 바이트)

```
오프셋  크기   필드                 설명
──────────────────────────────────────────────────────────────
0x00    4B     page_type            PAGE_TYPE_LEAF (0x03)
0x04    4B     parent_page_id       부모 내부 노드 (루트이면 0)
0x08    4B     key_count            저장된 엔트리 수
0x0C    4B     next_leaf_page_id    오른쪽 형제 리프 (0=없음)
0x10    4B     prev_leaf_page_id    왼쪽 형제 리프 (0=없음)
──────────────────────────────────────────────────────────────
합계: 20 바이트
```

### 리프 엔트리 (14 바이트)

```
오프셋  크기   필드        설명
────────────────────────────────────────────
0x00    8B     key         인덱스 키 (id 값)
0x08    4B     page_id     행이 저장된 힙 페이지 ID
0x0C    2B     slot_id     힙 페이지 내 슬롯 번호
────────────────────────────────────────────
합계: 14 바이트 (= 8 + sizeof(row_ref_t))
```

### 용량 계산

```
사용 가능 공간 = 4096 - 20(헤더) = 4076 바이트
최대 엔트리   = 4076 / 14 = 291 개
분할 기준     = 291 / 2 = 145 개 (절반을 새 페이지로)
```

## 4. B+ tree 내부 노드 페이지 (PAGE_TYPE_INTERNAL)

자식 노드로의 라우팅 정보를 저장합니다. 실제 데이터(row_ref) 는 없고, key 와 자식 page_id 만 있습니다.

```
오프셋 0                                                     오프셋 4096
┌──────────────┬──────────┬──────────┬──────────┬────────────────────┐
│ 내부노드 헤더 │ entry 0  │ entry 1  │ entry 2  │     빈 공간        │
│     16B      │   12B    │   12B    │   12B    │                    │
└──────────────┴──────────┴──────────┴──────────┴────────────────────┘
```

### 내부 노드 헤더 (16 바이트)

```
오프셋  크기   필드                      설명
──────────────────────────────────────────────────────────────
0x00    4B     page_type                 PAGE_TYPE_INTERNAL (0x04)
0x04    4B     parent_page_id            부모 노드 (루트이면 0)
0x08    4B     key_count                 저장된 엔트리 수
0x0C    4B     leftmost_child_page_id    가장 왼쪽 자식 페이지 ID
──────────────────────────────────────────────────────────────
합계: 16 바이트
```

### 내부 노드 엔트리 (12 바이트)

```
오프셋  크기   필드                  설명
────────────────────────────────────────────────
0x00    8B     key                   구분 키 (separator key)
0x08    4B     right_child_page_id   이 key 이상인 값이 있는 자식
────────────────────────────────────────────────
합계: 12 바이트
```

### 라우팅 규칙

```
내부 노드: leftmost_child=P0, entries=[(K1,P1), (K2,P2), (K3,P3)]

검색 key 에 따라:
  key < K1       → P0 (leftmost_child)
  K1 ≤ key < K2  → P1
  K2 ≤ key < K3  → P2
  key ≥ K3       → P3
```

## 5. 빈 페이지 (PAGE_TYPE_FREE)

삭제된 페이지의 재활용을 위한 연결 리스트 노드입니다.

```
오프셋 0              오프셋 8                          오프셋 4096
┌─────────────────────┬────────────────────────────────────────────┐
│  page_type (4B)     │                                            │
│  next_free_page(4B) │            나머지 전부 0                   │
└─────────────────────┴────────────────────────────────────────────┘
합계: 8 바이트만 사용, 나머지 4088 바이트는 0
```

할당 시 head 에서 꺼내고(LIFO), 해제 시 head 에 삽입합니다.

## 6. 페이지 타입별 비교 요약

```
타입       헤더크기  엔트리크기  최대엔트리  핵심 역할
─────────────────────────────────────────────────────────
HEADER     676B      -          -          DB 전역 메타데이터
HEAP       16B       8B(슬롯)   ~78행      실제 행 데이터 저장
LEAF       20B       14B        291개      key → 힙 위치 매핑
INTERNAL   16B       12B        340개      자식 노드 라우팅
FREE       8B        -          -          빈 페이지 재활용
```

이 다섯 가지 페이지 타입이 조합되어 minidb 의 디스크 저장 레이어를 이룹니다. 첫 4 바이트의 태그 하나로 한 블록을 다섯 가지 의미 중 하나로 해석할 수 있다는 점이 디스크 기반 엔진 설계의 골격입니다.
